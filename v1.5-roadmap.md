# Genm-codex `v1.5` Roadmap

## 目的

`v1.5` 不再延续 `v1.4` 的“主线基础设施补洞”节奏，而是进入：

1. 消费层产品化
2. 样本库与回归治理
3. profile / bucket / tagpack 的 registry 化
4. `novel-scan` 的 contract 化与 consumer 接线

重点不是“再堆能力”，而是：

- 把已有能力变成稳定消费面
- 把扩面工作交给 `mimo` 时，先把 contract / guardrail / verify 门定死
- 明确哪些东西在 `v1.5` 只做准备，不做最终实现

---

## 当前基线

`v1.4` 主线已完成，当前主仓基线：

- `workflow-health / quality-audit / knowledge-projection`
- `project status dashboard`
- P0 + 非 P0 bucket overlay 基础接线
- `e2e-novel` 与《嫁妆单》真实 smoke / regression 样本
- `state schema / template / docs / tests` 基本闭环

## 总判断

`v1.5` 里最该优先做的，不是直接大扩面，而是先完成一组 **codex-first 前置任务**，把后续能交给 mimo 的范围做实。

也就是说：

- `profile` 可以扩
- `novel-scan` 可以推进
- `skill merge` 可以做准备
- `runtime/plugin` 可以做边界研究

但这些都不该在没有 contract / verify / non-goal 的情况下直接开做。

当前这份 roadmap 对“方向判断”已经够用，但如果要真正开始分工执行，还必须明确三件事：

1. `codex-first` 任务各自要产出什么文件
2. 每个前置任务跑什么验证才算完成
3. 完成后，`mimo` 是被解锁了哪一组任务

下面的 `P0` 已按这个标准补齐。

## P0：Codex-First 前置任务

这组任务优先级最高。它们做完前，不建议 mimo 大规模扩面。

### P0-A. Sample Manifest Contract

- Status: `[done]`
- Owner: `[codex]`
- Start: `[codex-first]`
- 目标：
  - 定义样本清单的统一 contract
  - 明确 `smoke / regression / 高置信` 三层样本的字段和准入条件
- 输出：
  - `shared/templates/sample-manifest-v1.json` 或等价 schema
  - 样本分层规则文档
- Verify：
  - `pytest -q tests/test_real_regression_samples.py tests/test_real_e2e_novel_smoke.py`
  - 新增 sample manifest contract 对应测试
- 解锁：
  - `P2-B`
- 为什么要先做：
  - 不先定 contract，`mimo` 扩样本库会继续靠文档口径，难以自动消费

### P0-B. Profile Expansion Contract

- Status: `[done]`
- Owner: `[codex]`
- Start: `[codex-first]`
- 目标：
  - 明确新 profile 的最低 contract
  - 明确何时必须配 `platform_positioning`
  - 明确何时必须配 `bucket overlay`
  - 明确 profile 扩面必须过哪些验证
- 输出：
  - profile 扩面规则文档
  - profile 校验测试扩展
- Verify：
  - `pytest -q tests/test_profile_contract.py tests/test_content_positioning.py`
  - `bash scripts/validate-migration.sh`
- 解锁：
  - `P2-A`
- 为什么要先做：
  - 不先定门，`mimo` 批量扩 profile 很容易再次出现 YAML 漂移和 overlay 不一致

### P0-C. Scan Result Contract

- Status: `[done]`
- Owner: `[codex]`
- Start: `[codex-first]`
- 目标：
  - 定义 `novel-scan` 输出 sidecar 的稳定字段
  - 明确 `confidence`、`report-only`、`project-annotate` 的边界
  - 明确哪些 consumer 能读、读什么、不能覆盖什么
- 输出：
  - scan sidecar contract
  - consumer 读取顺序说明
- Verify：
  - `pytest -q tests/test_scan_result_contract.py tests/test_novel_scan.py`
  - 至少一条 report-only smoke 样本通过
- 解锁：
  - `P2-C`
- 为什么要先做：
  - 不先定 contract，scan 很容易重新变成“弱证据直接污染 workflow”

### P0-D. Skill Rationalization Policy

- Status: `[done]`
- Owner: `[codex]`
- Start: `[codex-first]`
- 目标：
  - 定义哪些 skill 允许合并
  - 哪些只能做 alias / deprecation
  - 哪些是 protected，不允许在 `v1.5` 动
- 输出：
  - skill rationalization policy
  - merge-map contract
- Verify：
  - overlap inventory 与 policy 的 protected / alias / merge 分类可以映射到现有 skill 清单
- 解锁：
  - `P2-D`
- 为什么要先做：
  - 不先定边界，`31 -> 25` 会变成大范围破坏性调整

### P0-E. Runtime Boundary ADR

- Status: `[done]`
- Owner: `[codex]`
- Start: `[codex-first]`
- 目标：
  - 明确 `plugin/runtime/daemon` 什么能研究、什么能试验、什么不能实现
  - 固定 `repo truth + sidecar + MCP` 的边界
- 输出：
  - 一份 ADR / boundary record
- Verify：
  - ADR 明确：
    - 允许研究什么
    - 禁止实现什么
    - 什么会破坏 `repo truth + sidecar + MCP`
- 解锁：
  - `P2-E`
- 为什么要先做：
  - 否则 `mimo` 做 runtime 研究时容易把方向写偏到新真值中心
