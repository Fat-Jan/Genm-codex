# 空挂 / 依赖现实盘点执行清单 v1

## 0. 任务目标

对 `Genm-codex` 仓库做一轮依赖现实盘点，找出：

1. 哪些 `shared/docs` 资产被真实消费
2. 哪些资产主要只是被 `SKILL.md` 或 docs 声明
3. 哪些资产高疑似空挂
4. 哪些依赖关系可能属于“依赖幻觉”

注意：

- 本任务是盘点与初筛，不是重构任务
- 本任务不是删除任务
- 本任务不是脚本改造任务

---

## 1. 执行范围

第一轮只允许扫描以下区域，不要全仓铺开。

### 1.1 声明入口

重点看：

- `skills/novel-review/SKILL.md`
- `skills/novel-outline/SKILL.md`
- `docs/00-当前有效/shared-asset-dependency-map.md`

### 1.2 资产范围

重点只盘下面三类：

#### A. `shared/templates/` 下被声明或看起来重要的文件

例如：

- `*.json`
- `*.schema.json`
- `*.md`

#### B. `shared/references/` 下被声明引用的文件

例如：

- `shared/*.md`
- `commands/*.md`
- `writing/*.md`

#### C. 少量 docs 框架文件

仅限在 `novel-review` / `novel-outline` 中被显式提到的 docs 文件，不要把整个 `docs/` 全扫一遍。

### 1.3 暂不处理

第一轮先不要深挖这些：

- `shared/profiles/` 全量目录
- 所有 `bucket-*.yaml`
- 所有 `profile-tomato.yaml`
- 所有项目样本目录
- 全量 archived docs

原因：这批体量太大，容易把第一轮盘点做散。

## 3. 每个候选资产必须输出的字段

每个文件都要输出以下字段：

| 字段 | 含义 |
| --- | --- |
| `path` | 文件路径 |
| `declared_in` | 在哪些文件里被声明 |
| `consumed_in` | 在哪些脚本 / 测试 / 技能 / docs 里被实际命中 |
| `declaration_types` | 声明类型 |
| `evidence_level` | 证据等级 |
| `consumption_guess` | 初步消费分类 |
| `notes` | 简短说明 |

### 3.1 `declared_in`

列出声明位置，格式尽量包含文件和行号。

例如：

- `skills/novel-review/SKILL.md:L36-L37`
- `docs/00-当前有效/shared-asset-dependency-map.md:L13-L20`

### 3.2 `consumed_in`

列出实际命中位置，按类型分组：

- `scripts`
- `tests`
- `skills`
- `docs`
- `findings`

如果没有命中，也要明确写：

- `none found`

### 3.3 `declaration_types`

只允许使用以下值：

- `required-like`
- `conditional-like`
- `reference-like`
- `map-listed`

解释：

#### `required-like`

文案看起来像“必须读 / 先读 / 主依赖”。

#### `conditional-like`

文案看起来像“按 genre / 情况读”。

#### `reference-like`

更像补充说明或参考。

#### `map-listed`

主要来自 dependency map 里列出。

### 3.4 `evidence_level`

只允许使用以下值：

- `E0`
- `E1`
- `E2`
- `E3`
- `E4`

判定规则见后文。

### 3.5 `consumption_guess`

只允许使用以下值：

- `hard-consumed-candidate`
- `soft-consumed-candidate`
- `declared-only-candidate`
- `orphan-candidate`

注意：这里只能给 `candidate`，不能下最终结论。

## 4. Evidence 判定规则

这个部分必须严格执行，不能自由发挥。

### E0：存在

条件：

- 文件存在
- 没找到任何声明或命中

### E1：提及

条件：

- 只在 docs / findings / 阶段文档中被提及
- 没有技能声明、脚本消费、测试消费证据

### E2：声明

条件：

- 在 `SKILL.md` 或 dependency map 中被显式声明
- 但没找到脚本 / tests 直接消费证据

### E3：消费

条件：

- 被脚本直接读取 / 匹配 / 使用
- 或被测试直接校验
- 或在技能主流程中有更强的消费证据

### E4：结果影响

条件：

- 不仅被消费，而且明显影响校验结果、输出结果或主流程行为

注意：如果拿不准 `E3` 还是 `E4`，先保守打 `E3`。

---

## 5. Consumption Guess 判定规则

### `hard-consumed-candidate`

满足其一即可：

- 被 script 直接读取
- 被 tests 明确校验
- 被多个强消费点命中
- 明显影响运行结果

### `soft-consumed-candidate`

满足其一即可：

- 主要在 `SKILL.md` 中被要求阅读
- 主要作为流程 / 提示 / 参考文档出现
- 找不到程序硬消费，但技能叙事依赖明显

### `declared-only-candidate`

满足其一即可：

- 在 `SKILL.md` / dependency map 中被写得很重要
- 但找不到更硬的消费证据
- 更像“有依赖主张，无闭环证据”

### `orphan-candidate`

满足其一即可：

- 基本没有声明
- 也没发现消费
- 或只有很弱的历史提及
- 且看不出它仍在主链上起作用

---

## 6. 执行时的命中类型权重

为了避免“命中次数幻觉”，需要记住命中类型权重。

### 权重从强到弱

1. `scripts` / `tests`
2. `skills` 主声明
3. `dependency map`
4. `docs` 说明
5. `findings` / archived plans

解释：

- findings 命中再多，也不等于核心依赖
- script 直接读取 1 次，通常比 docs 提到 10 次更重要

---

## 7. 禁止事项

### 禁止 1：禁止直接给删除建议

不能输出：

- 删除这个文件
- 清理这个目录
- 改路径
- 改脚本读取逻辑

只能输出：

- `orphan-candidate`
- `review-needed`

### 禁止 2：禁止自己发明新分类

只能使用本清单定义的字段和值。

### 禁止 3：禁止因为文件名听起来重要，就直接判核心

例如：

- `schema`
- `registry`
- `contract`
- `map`

这些名字都不能自动等于 `hard-consumed`。

### 禁止 4：禁止把 docs 提及当成真实消费

`docs/findings` 命中只算 `E1` 或辅助说明。

### 禁止 5：禁止把全仓所有 shared 文件都拉进来

只处理：

- 声明入口抽出来的候选项
- 少量补充核心候选

## 8. 输出格式要求

建议输出成表格或 JSON 风格清单。

如果是表格，建议使用：

| path | declared_in | consumed_in | declaration_types | evidence_level | consumption_guess | notes |
| --- | --- | --- | --- | --- | --- | --- |

### `notes` 应怎么写

只写一句短说明，例如：

- `在 SKILL.md 中多次声明，但未见 scripts/tests 直接消费`
- `被 profile_contract.py 直接读取，属于强消费候选`
- `主要作为提示性文档出现，倾向 soft-consumed`
- `仅见历史提及，当前疑似空挂`

不要写大段分析。

## 11. 复检重点

执行结果回来后，重点复检以下几类：

1. 被判成 `orphan-candidate` 的文件，防止误伤 `soft-consumed`
2. 被判成 `hard-consumed-candidate` 的 `schema / registry / map`，防止因为文件名误判
3. `evidence_level = E2` 但声明很重的项，这类最容易属于 dependency illusion
4. 多个 `SKILL.md` 都声明，但脚本不读的项，这类通常最值得重写依赖叙事
