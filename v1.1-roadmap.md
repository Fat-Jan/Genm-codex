# Genm-codex `v1.1` Roadmap

## 目的

这份 roadmap 是 `Genm-codex` 下一阶段工作的**根目录主计划**。

它回答：

- `v1.0.0` 之后最值得继续推进的方向是什么
- 哪些工作已经完成，哪些还没开始
- 哪些属于主线，哪些只是补线或暂缓项

它是当前 `v1.1` 相关工作的**单一真源**。

## 状态说明

- `[done]`：已完成，当前不再是 blocker
- `[in_progress]`：已开始，仍在推进
- `[planned]`：已确认要做，但尚未开始
- `[blocked]`：依赖上游收口后再做
- `[deferred]`：当前明确延后，不进本轮主线

## 当前总判断

- 当前更合理的形态不是“继续加很多新 skill”
- 而是：
  1. 收紧 `profile` / `bucket` / `tagpack` / `state` 的 contract
  2. 把默认工作流里仍然半联动的能力真正接上线
  3. 再用外部资料做内容层校准

## 已完成前置收口

这些不是 `v1.1` 主线本身，但已经完成，可以作为后续推进的前置基础。

- `[done]` `shared` governance 收口
  - `shared/sync-governance.json`
  - governance-aware `sync-shared-from-genm.sh`
  - drift block / protected local restore
- `[done]` `workflow_state` maintenance 尾段机械闭环
  - maintenance 完成后真实写回 `.mighty/workflow_state.json`
  - `current_step -> snapshot`
  - `last_successful_checkpoint -> maintenance`
- `[done]` `active-context` 收回到摘要/指针层
  - 不再复制 `recent_guardrails` 正文
- `[done]` 当前有效层文档和治理台账收口

## 主线 A：Profile Contract 升级

### A1. 统一 profile schema

- Status: `[done]`
- 目标：
  - 统一 `shared/profiles/*/profile.yaml` 的核心字段结构
  - 先解决异构字段命名问题，再谈内容升级
- 重点文件：
  - `shared/profiles/*/profile.yaml`
  - `shared/profiles/README.md`
  - `skills/novel-genre/SKILL.md`

### A2. 明确 `profile -> state.genre_profile` 投影契约

- Status: `[done]`
- 目标：
  - 明确哪些 profile 字段允许写入 `.mighty/state.json`
  - 继续保持轻量投影，不把 profile 全量塞进 state
- 重点文件：
  - `shared/templates/state-v5-template.json`
  - `shared/references/shared/state-schema.md`
  - `skills/novel-genre/SKILL.md`

### A3. 拆分 `core profile / platform overlay / bucket overlay / references`

- Status: `[done]`
- 目标：
  - `profile` 做题材基础层
  - `platform overlay` 做平台差异
  - `bucket overlay` 做番茄内容池差异
  - 长文本方法论回到 `docs/` 或 `shared/references/`
- 重点文件：
  - `shared/profiles/*`
  - `docs/fanqie-content-buckets.md`
  - `docs/fanqie-bucket-constraints.md`
- 当前结果：
  - 已建立分层 contract 与解析规则
  - 已支持：
    - `profile.yaml`
    - `profile-<platform>.yaml`
    - `bucket-<bucket>.yaml`
    - 目录内 `*.md` reference files
  - 历史 embedded 长文本当前视为 legacy reference，而不再算 core contract

### A4. 统一 consumer 入口

- Status: `[done]`
- 目标：
  - 优先由 `novel-genre` 负责 profile 解析
  - 其他 skill 更少直接吃异构原始 profile
- 重点技能：
  - `novel-init`
  - `novel-genre`
  - `novel-outline`
  - `novel-write`
  - `novel-review`
  - `novel-package`
- 当前结果：
  - 已统一 `novel-init / novel-genre / novel-outline / novel-write / novel-review / novel-package` 的 profile 消费口径
  - 统一原则为：
    - `novel-genre` / `state.genre_profile` 是主入口
    - raw profile 只在缺少额外细节时通过 `scripts/profile_contract.py` 解读
    - raw profile 按 `core profile -> platform overlay -> bucket overlay -> reference files` 顺序消费
    - legacy embedded 长文本不再视为 authoritative core config

## 主线 B：默认工作流补线

### B1. `fetch MCP` provider registry / workspace config

- Status: `[done]`
- 当前状态：
  - `acquire_source_text.py` 已支持 `--fetch-cmd` / `GENM_FETCH_MCP_CMD`
  - 但仓库没有 workspace 级 provider 声明，也没有统一 registry
- 目标：
  - 明确“已配置 / 未配置 / 当前实际 provider”
  - 减少 fetch 能力只存在于 CLI 参数层的半联动状态
- 当前结果：
  - 已新增 provider registry
  - `acquire_source_text.py` 现在支持：
    - project `.mighty/config.json`
    - `~/.codex/config.toml`
    - provider diagnostics
    - `--show-provider-config`

### B2. `MCP memory` project-local mapping

- Status: `[done]`
- 当前状态：
  - 边界已定义：只做协调记忆，不做 canon 真源
  - 但还没有真正的 project-local mapping
- 目标：
  - 只同步治理摘要、恢复建议、阶段状态
  - 不同步 canon / sidecar 正文
- 当前结果：
  - 已新增 `build_memory_context.py`
  - 默认维护链现在会生成 `.mighty/memory-context.json`
  - memory payload 只保留 safe summary，不直接复制 canon / prose / sidecar 正文

### B3. `launch-stack` auto-compile 触发点

- Status: `[done]`
- 当前状态：
  - `novel-init` 只写 `preselect` placeholder
  - 番茄项目仍依赖 `novel-outline` 后手动运行 `fanqie_launch_stack.py`
- 目标：
  - 增加守卫式 auto-compile 提醒或触发点
  - 避免番茄项目经常停在 placeholder
- 当前结果：
  - `setting gate(outline)` 现在会在番茄项目中守卫式 auto-compile `launch-stack`
  - 已有 locked sidecar 时不会重复覆盖

### B4. `snapshot` owner 收口

- Status: `[done]`
- 当前状态：
  - `workflow_state` 已推进到 `snapshot`
  - 但 snapshot artifact 还没自动生成
- 目标：
  - 明确谁负责生成 snapshot
  - 把 `snapshot` 从“名义上的下一步”变成真实尾段
- 当前结果：
  - 已新增 `generate_snapshot.py`
  - 默认维护链尾段现在会真实生成 snapshot artifact
  - `workflow_state` 会以 `snapshot` 作为最终 checkpoint 收口

### B5. `novel-log` 默认 trace writer / init path

- Status: `[done]`
- 当前状态：
  - `novel-log` 只有读链
  - 默认工作流缺统一 trace writer
- 目标：
  - 决定要不要上默认轻量 trace logging
  - 若不上，则明确降为专家能力
- 当前结果：
  - 已新增 `trace_log.py`
  - `setting gate`、`maintenance`、`snapshot`、`memory-context` 现在会写入轻量 trace log
  - `novel-log` 不再只适用于“完全手工初始化日志的项目”

## 主线 C：内容与平台校准

  - 已用番茄官方作者侧资料校准：
    - profile core 应承载的内容
    - packaging / outline / write / review 的使用口径
    - 不应继续膨胀进 profile core 的长文本类型
  - 当前完成的是：
    - **4A：校准口径收口**
    - **4B：首批优先 profile 内容实改**
  - 当前首批优先范围：
    - `palace-intrigue`
    - `realistic`
    - `xuanhuan/profile-tomato`

### C2. bucket 与题材映射校准

- Status: `[done]`
- 目标：
  - 让 `profile / bucket / tagpack` 的关系不再只靠零散文档和 skill 文案维持

  - 已明确：

    - `tone_guardrails`

  - 已把组合题材 / 多主题卖点 / 群像结构模式纳入映射校准范围
  - 已新增机器映射文件：
    - `shared/templates/content-positioning-map-v1.json`

## 当前明确暂缓项

  - 当前不阻塞默认主线

- Status: `[deferred]`
- 原因：

- Status: `[deferred]`

## 推荐执行顺序

1. `[planned]` 主线 A：Profile Contract 升级
2. `[planned]` 主线 B：默认工作流补线
3. `[blocked]` 主线 C：内容与平台校准
4. `[deferred]` 继续维持实验边界，不额外扩主线范围

## 执行编排

### Phase 1：先串行收 `Profile Contract`

- Status: `[planned]`
- 先做：
  - `A1. 统一 profile schema`
  - `A2. 明确 profile -> state.genre_profile 投影契约`
  - `A3. 拆分 core profile / platform overlay / bucket overlay / references`
- 原因：
  - 后续 consumer 收口和内容校准都依赖这层 contract

### Phase 2：并行补线

- Status: `[planned]`
- 可以并行的两组：

#### 2A. 默认工作流补线

- `B3. launch-stack auto-compile 触发点`
- `B4. snapshot owner 收口`
- `B5. novel-log 默认 trace writer / init path`

#### 2B. MCP / fetch 接线

- `B1. fetch MCP provider registry / workspace config`
- `B2. MCP memory project-local mapping`

- 原因：
  - 这两组都不应该反向决定 profile contract
  - 但它们都需要在 consumer 收口前定型

### Phase 3：再串行做 consumer 收口

- Status: `[done]`
- 重点对象：
  - `novel-init`
  - `novel-genre`
  - `novel-outline`
  - `novel-write`
  - `novel-review`
  - `novel-package`
- 原因：
  - 要等 Profile Contract 和补线策略都稳定后，再统一接入口

  - 组合题材 / 多主题卖点 / 群像结构模式只在 Phase 3 按设计稿接入
  - 接入时应遵循：
    - 主 profile
    - 主 bucket

    - 风味护栏
  - 参考：
    - `docs/superpowers/specs/2026-03-26-composite-genre-phase3-integration-design.md`
- 当前结果：
  - 已补 `genre_profile` 的组合题材轻量字段：
    - `tagpacks`
    - `strong_tags`
    - `narrative_modes`
    - `tone_guardrails`
    - `positioning_sidecar`

  - 已接入：

    - `novel-write`
    - `novel-review`

### Phase 4：最后再做外部资料校准

- Status: `[done]`
- 先决条件：
  - Phase 1 完成
  - Phase 2 / 3 至少完成主干
- 内容：

  - `C2. bucket 与题材映射校准`
- 原因：
  - 如果在 contract 未稳定前就先做外部内容升级，后面很容易返工
- 当前结果：
  - 已完成 **Phase 4A：当前有效口径收口**
  - 已完成 **Phase 4B：首批优先 profile 内容升级 / 映射实改**
  - 外部资料校准结论已落在 current-effective 文档、机器映射文件与 roadmap 中

## 验证策略

- 结构变更优先用：
  - `pytest -q`
  - `bash scripts/validate-migration.sh`
- workflow / shared / context 变更优先补 targeted tests
- profile 内容校准阶段优先：
  - 官方资料比对
  - 真实项目 smoke
  - 小范围回写验证

## 不做什么

- 不新开一份平行“二期计划”文档
- 不把 MCP 直接变成项目真值层
- 不在 profile 结构未统一前就盲目扩 profile 内容
- 不把实验能力误包装成默认主线
