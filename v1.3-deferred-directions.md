# Genm-codex `v1.3` Deferred Directions Reference

## 目的

这份文档记录当前**不进入 `v1.3` 主线完成条件**，但仍值得保留为后续优化参考的方向。

它回答 3 个问题：

- 这些方向分别能带来什么效果
- 为什么当前阶段不建议把它们拉进默认主线
- 未来出现什么信号时，值得重新拉起

## 读取方式

- `可选并行实验`：可以继续推进，但不计入 `v1.3` 主线完成条件
- `暂不拉主线`：方向本身有价值，但当前不应进入主线排序
- `当前不建议启动`：本阶段启动成本和干扰明显大于收益

## 当前总判断

`v1.3` 仍应优先解决：

1. 默认工作流稳定性与恢复体验
2. 上游结构合同定义
3. 质量路由一致性
4. 高频 consumer 的规则装载与读法一致性

因此，下列方向当前更适合作为：

- 参考池
- 实验支线
- 未来阶段候选

而不是本轮默认主线。

---

## 1. `novel-scan` 深化为默认主线

- 当前分类：`可选并行实验`
- 当前结论：可以继续做，但不要升格为默认主线

### 它的作用 / 效果

- 为项目补外部 research 与市场趋势候选
- 给 `setting gate`、`review-sync-queue`、包装判断提供额外信号
- 在 report-only 或 candidate-handoff 路径下，帮助识别值得人工复核的方向

### 为什么现在不拉进默认主线

- 它依赖外部证据，时效性、可信度、可重复性都弱于本地 truth 文件
- 一旦进入默认主线，用户很容易把外部候选误当成 canon
- 当前更适合把它当作：
  - 候选 research 输入
  - confidence-aware 提示源
  - `gate -> review-sync-queue` 的辅助链

### 现在适合做的部分

- 继续加强：
  - `report-only`
  - `candidate-handoff`
  - `confidence-aware` 输出
  - 与 `setting gate` 的候选协同
- 继续避免：
  - 直接改写 `设定集/`
  - 绕过本地 truth gate
  - 自动把 scan 结果升级为 canon

### 什么信号出现后再考虑升主线

- 2-3 条真实样本证明它能稳定提升判断质量，而不是制造噪音
- 输出具备稳定 confidence、来源追踪和可复核路径
- 能证明它不会破坏：
  - `setting gate`
  - 本地真值
  - 默认写作链

## 2. `monolithic runtime / plugin system`

- 当前分类：`当前不建议启动`
- 当前结论：不进入当前主线

### 它的作用 / 效果

- 把现在分散的：
  - `docs`
  - `skills`
  - `scripts`
  - `sidecars`
  统一到一个总调度层
- 让 hook、流程编排、插件扩展、运行时治理更集中

### 为什么现在不做

- 当前主问题不是“缺一个总 runtime”，而是很多 contract 还在收口
- 如果现在先造统一 runtime，本质上是在冻结一套还没完全稳定的边界
- 实现和迁移成本都高，但对普通写作者的直接收益并不成比例

### 什么信号出现后再考虑

- `B + F + C` 收口后，仍然大量出现重复编排逻辑
- 现有 `scripts + sidecars` 的编排复杂度已经明显压过文档/skill 可维护边界
- 能明确证明这是在减少复杂度，而不是换一种方式堆复杂度

---

## 3. 把 `MCP memory` 升成项目真值层

- 当前分类：`当前不建议启动`
- 当前结论：只允许做协调记忆，不允许升格为 canon 真源

### 它的作用 / 效果

- 提升跨会话连续性
- 记录最近关注问题、恢复提示、协作摘要
- 理论上可增强跨轮检索与状态衔接

### 为什么现在不做

- 会制造第二真值中心
- 一旦 memory 与本地文件不一致，冲突裁决、审计、回滚都会立刻变重
- 当前项目已经明确：
  - 本地文件才是唯一真值
  - memory 只能做协调层

### 现在适合做的部分

- 继续记录：
  - 任务摘要
  - 恢复提示
  - 非 canon 的协作线索
- 不记录：
  - `state.json` 全量镜像
  - `设定集/` 全量 canon
  - sidecar 正文

### 什么信号出现后再考虑

- 有严格的 projection schema
- 有明确的冲突裁决规则
- 有可审计、可重建、可回滚路径
- 且仍能保持“本地文件唯一真值”的前提

## 4. 把 `novel-log` 拉回默认主线

- 当前分类：`暂不拉主线`
- 当前结论：继续保留为专家辅助能力

### 它的作用 / 效果

- 暴露 trace、步骤和运行证据
- 对复杂 workflow 排障、恢复和诊断很有帮助

### 为什么现在不拉回主线

- 默认用户应先通过：
  - `novel-status`
  - `novel-resume`
  - `novel-workflow`
  理解项目状态
- 如果必须依赖 trace log 才能理解流程，说明默认 UX 还没收口
- 把 log 前置为主线能力，会让内部诊断层变成普通用户噪音

### 什么信号出现后再考虑

- 大量真实问题都证明：
  - `status/resume/workflow` 不足以表达事务状态
  - trace log 已经成为常规恢复必需品
- 并且能把 log 提升为用户可读视图，而不是原始调试输出

## 5. 脱离用户价值的纯治理 registry / manifest 工程

- 当前分类：`暂不拉主线`
- 当前结论：只在明确服务 consumer contract 时做，不单独起主线

### 它的作用 / 效果

- 集中维护 bundle、读档规则、sidecar 入口、规则装载协议
- 提升维护者视角下的一致性

### 为什么现在不单独做

- 如果脱离 `outline / write / review / package` 的真实消费场景，它很容易变成“架构看起来更整齐”，但用户价值不明显
- 当前值得做的是：
  - `C1` 这种直接服务高频 consumer 的 manifest 化
- 当前不值得做的是：
  - 只为治理而治理
  - 只为“像架构”而起 registry 主线

### 什么信号出现后再考虑扩大范围

- 高频 consumer 的 contract 已稳定
- 已能证明 manifest / registry 在减少回归和漂移，而不是新增一层维护负担
- 新规则层的引入频率足够高，集中治理的收益开始明显超过成本

---

## 当前建议

后续若要从这份参考文档里重新拉起某一项，优先顺序建议是：

1. 先看它是否服务当前主问题
2. 再看它是否会制造第二真值或重复 contract
3. 最后再看它的工程成本是否值得本轮承担

当前最适合继续保留活性的只有一项：

- `novel-scan` 作为**可选并行实验**

其余几项更适合作为：

- 边界提醒
- 架构决策记录
- 下一阶段候选池
