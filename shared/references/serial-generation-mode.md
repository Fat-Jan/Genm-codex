# 串行生成模式

**版本**: v1.0
**创建日期**: 2026-03-18
**适用范围**: Genm 小说生成系统

---

## 1. 模式概述

### 1.1 串行 vs 并行

| 维度 | 并行生成 | 串行生成 |
|------|---------|---------|
| **执行方式** | 多章节同时生成 | 逐章顺序生成 |
| **上下文依赖** | 弱（各章节独立） | 强（前章决定后章） |
| **连贯性保证** | 依赖概要描述（弱） | 依赖真相文件（强） |
| **生成速度** | 快（并行） | 慢（顺序） |
| **适用场景** | 独立内容、批量草稿 | 小说、强依赖内容 |

### 1.2 为什么小说必须串行

**小说的本质特征**:

- **强时间线**: 事件按时间顺序发生，后章依赖前章结果
- **状态累积**: 角色修为、关系、资源随章节递进
- **伏笔回收**: 前文埋下的伏笔需要后文呼应
- **情感弧线**: 角色情感变化是连续过程

**并行生成的问题**（已在 Phase 1-2 验证）:

```
第3章 Agent → "主角炼气五层"
第4章 Agent → "主角一天一夜突破"（未读取第3章，重复突破）
结果: 修为描述矛盾，读者困惑
```

**串行生成的优势**:

```
第3章 → 更新 current_state.md（修为: 五层） → 第4章读取（确认: 五层）
结果: 状态一致，逻辑连贯
```

### 2.1 标准流程

```
┌─────────────────────────────────────────────────────────────────┐
│                      串行生成流程图                               │
└─────────────────────────────────────────────────────────────────┘

    开始
      │
      ▼
┌─────────────┐
│ 读取真相文件 │ ◄── 必须读取 truth/ 全部7个文件
│  (Queen Bee) │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  生成第N章   │
│  (Writer)   │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  质量检查    │
│(Worker Bees)│
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 更新真相文件 │ ◄── 立即更新相关真相文件
│  (Hive Bee) │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  创建快照    │ ◄── 保存当前状态

│  还有下一章? │ ─────────► 结束
│             │
└──────┬──────┘
       │ 是
       ▼
   回到"读取真相文件"
```

### 2.2 真相文件驱动机制

**生成前必须读取**（强制）:

| 文件 | 用途 | 关键信息 |
|------|------|---------|
| `current_state.md` | 当前状态 | 修为、位置、进行中的事件 |
| `character_matrix.md` | 人物关系 | 角色状态、关系变化 |
| `pending_hooks.md` | 待回收伏笔 | 未解决的伏笔列表 |
| `emotional_arcs.md` | 情感弧线 | 角色情感轨迹 |
| `subplot_board.md` | 支线进度 | 各支线完成情况 |
| `particle_ledger.md` | 资源账本 | 功法、丹药、法宝变动 |
| `chapter_summaries.md` | 章节摘要 | 前章关键事件 |

**生成后必须更新**（强制）:

```yaml
更新清单:
  current_state.md:
    - protagonist.cultivation  # 修为变化
    - protagonist.location     # 位置变化
    - ongoing_events          # 事件进度

  character_matrix.md:
    - relationships           # 关系变化
    - status_changes          # 状态变化

  pending_hooks.md:
    - new_hooks               # 新增伏笔
    - resolved_hooks          # 回收的伏笔

  chapter_summaries.md:
    - new_chapter_summary     # 本章摘要
```

## 3. 连贯性保证策略

### 3.1 状态一致性检查点

**每章生成前检查**:

```yaml
检查点:
  - 主角当前修为: "炼气期五层"
  - 当前位置: "外门修炼室"
  - 进行中的事件: "准备外门大比"
  - 活跃角色: ["苏婉儿", "周通"]
  - 待回收伏笔: ["神秘老者", "上界秘密"]
```

**检查点不匹配处理**:

1. 暂停生成
2. 核对真相文件
3. 修正或标记异常
4. 继续生成

### 3.2 跨章节引用验证

**引用追踪表**（存储在 `chapter-index.json`）:

```json
{
  "chapter-003": {
    "references_to": ["chapter-001", "chapter-002"],
    "referenced_by": ["chapter-004", "chapter-005"],
    "key_events": ["突破五层", "击败周通"]
  }
}
```

**验证规则**:

- 引用的事件必须在被引用章节中真实发生
- 角色状态变化必须连续
- 时间线必须一致

### 3.3 伏笔回收机制

**伏笔生命周期**:

```
埋下伏笔 ──► 等待回收 ──► 回收完成 ──► 验证呼应
   │            │            │            │
   ▼            ▼            ▼            ▼
 pending     pending     resolved    verified
_hooks.md   hooks.md    _hooks.md    in audit
```

**回收检查清单**:

- [ ] 伏笔提出章节已标记
- [ ] 回收章节有明确呼应
- [ ] 回收时机合理（不过早/过晚）
- [ ] 回收方式符合读者预期

## 4. 半并行优化方案

### 4.1 两阶段生成

**阶段1: 串行生成详细章纲**（必须串行）

```
第1章大纲 ──► 更新真相 ──► 第2章大纲 ──► 更新真相 ──► ... ──► 第10章大纲
```

- 只生成大纲（每章~500字）
- 确保剧情连贯性
- 确定关键节点和伏笔位置

**阶段2: 基于章纲并行扩写**（可并行）

```
        ┌───► 扩写第1章 ───┐
        ├───► 扩写第2章 ───┤
章纲 ──► ├───► 扩写第3章 ───► 统一审核连贯性
        │      ...         │
        └───► 扩写第10章 ──┘
```

- 每章基于大纲扩写完整内容
- 关键：大纲必须足够详细，包含所有必要信息
- 并行扩写后统一审核

### 4.2 适用场景

| 场景 | 推荐模式 | 理由 |
|------|---------|------|
| 开篇黄金三章 | 纯串行 | 决定读者留存，必须精心打磨 |
| 关键剧情转折 | 纯串行 | 强依赖前文，状态变化密集 |
| 日常过渡章节 | 半并行 | 依赖较弱，可基于大纲并行 |
| 批量生成草稿 | 半并行 | 效率优先，后期统一润色 |

## 5. 使用指南

### 5.1 启用串行模式

**在 `/novel-write` 命令中**:

```markdown
## 模式选择

### 串行模式（推荐）
适合：正式写作、强剧情章节
特点：基于真相文件逐章生成，确保连贯性

### 半并行模式
适合：大纲到正文的批量扩写
特点：先串行大纲，再并行扩写

### 并行模式（不推荐小说）
适合：独立内容、测试草稿
特点：同时生成多章，可能产生连贯性问题
```

### 5.2 Queen Bee 工作流程调整

**标准串行流程**:

```yaml
workflow:
  name: "serial-chapter-generation"
  steps:
    - name: "read-truth-files"
      action: "read_all_files"
      path: "truth/"
      required: true

    - name: "generate-chapter"
      action: "generate_content"
      based_on: "truth-files"
      output: "chapter-draft"

    - name: "quality-audit"
      action: "run_audit"
      bees: ["consistency", "continuity", "pacing"]

    - name: "update-truth-files"
      action: "update_files"
      target: "truth/"
      changes: "{{chapter-changes}}"

    - name: "create-snapshot"
      action: "snapshot_state"
      path: ".mighty/snapshots/"

    - name: "next-chapter"
      action: "continue_if_has_more"
      goto: "read-truth-files"
```

### 5.3 故障恢复机制

**中断恢复**:

```bash
# 查看当前生成状态
/novel-status

# 输出:
当前章节: 5/10
状态: 中断（第5章生成中）
最后快照: chapter-004-v3

# 恢复生成（自动从快照恢复）
/novel-resume --from-snapshot=chapter-004-v3
```

**错误回滚**:

```bash
# 发现第5章有严重问题，回滚到第4章
/novel-rewrite --chapter=5 --reset-to=chapter-004-v3

# 重新生成第5章（基于第4章的真相文件）
/novel-write --chapter=5 --mode=serial
```

## 6. 性能优化

### 6.1 上下文管理

**只读必要内容**:

```
❌ 错误: 读取全部10章内容 (~40,000字)
✅ 正确: 只读真相文件 + 当前章索引 (~2,500字)
```

**真相文件缓存**:

- 在 session 中缓存真相文件内容
- 章节生成间隔短时不重新读取
- 文件修改时间变化时刷新缓存

### 6.2 增量更新

参见: [增量更新协议](./incremental-update-protocol.md)

```
传统方式: 更新完整文件 (2000 tokens)
增量方式: 只发送变更 (50-200 tokens)
节省: ~90%
```

## 7. 验证检查清单

### 7.1 生成前检查

- [ ] 已读取全部7个真相文件
- [ ] current_state.md 已更新到最新章节

### 7.2 生成后检查

- [ ] 已更新相关真相文件
- [ ] 已创建状态快照

- [ ] 角色关系变化已记录

### 7.3 连贯性验证

- [ ] 时间线连续无跳跃
- [ ] 修为/资源数值连续
- [ ] 角色出场状态一致

### 8.2 常见问题

**Q: 串行生成太慢，如何提速？**
A: 使用半并行模式，先串行大纲再并行扩写。或优化 LLM 调用参数。

**Q: 真相文件更新出错怎么办？**
A: 使用快照回滚到上一章，检查更新日志，修正后重新生成。

**Q: 如何处理回忆/倒叙章节？**
A: 即使时间线回溯，仍按章节顺序串行生成。在真相文件中标记时间线偏移。

### 8.3 术语表

| 术语 | 定义 |
|------|------|

| 伏笔回收 | 前文埋下的线索在后文得到呼应 |
| 章纲 | 章节详细大纲，包含关键事件和对话要点 |
