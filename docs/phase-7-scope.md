# Genm-codex 第七阶段范围设计

## 设计结论

前六阶段之后，剩余真正还值得进入主线的问题，已经不再是“再迁几个本地 skill”，而是两个高风险方向：

- **外部趋势扫描契约**
- **shared 资产治理**

这两者都不适合像前几阶段那样直接落地成普通本地 skill，因此第七阶段的重点是：

1. 先把边界和契约写清楚
2. 只在约束稳定后再进入实现

## 阶段划分

### Phase 7A：外部趋势扫描契约层

目标：

- 为 `novel-scan` 建立最小稳定契约

范围：

- 定义外部扫描依赖
- 定义数据来源可信度层级
- 定义本地结果落盘格式
- 定义“无网络 / 无外部服务”时的退化路径

不在这一阶段直接承诺：

- 稳定联网扫描实现
- 自动化 profile 权重回写

### Phase 7B：shared 资产治理层

目标：

- 评估 `shared` 资产是否还要继续 full-copy
- 设计 selective-sync 的演进路径

范围：

- 识别哪些 shared 资产变化频率高
- 识别哪些引用路径耦合最强
- 设计 selective-sync 的边界
- 定义迁移前置检查

不在这一阶段直接承诺：

- 立即切换 selective-sync
- 大规模路径重写

## 为什么现在才做

前六阶段主要解决的是：

- 把主体功能迁过来
- 把本地工作流跑顺
- 把用户入口收口

只有在这些都基本稳定之后，才适合碰：

- 外部依赖
- shared 资产治理

否则会把主线重新拖回不稳定状态。

## `novel-scan` 的当前定位

### 价值

- 高价值
- 能补平台趋势、题材热度、外部写作模式变化

### 风险

- 强依赖外部平台
- 强依赖联网能力
- 强依赖抓取结果质量
- 强依赖“是否允许自动回写 profile”

### 建议

先做契约设计，不急着实现默认可用 skill。

## selective-sync 的当前定位

### 价值

- 高价值
- 可以减少 `shared` 全量复制后的漂移压力

### 风险

- 会触及大量路径假设
- 会影响：
  - `novel-init`
  - `novel-write`
  - `novel-outline`
  - `novel-genre`
  - `novel-analyze`
  - `novel-precheck`
  等已迁移 skill 的 shared 路径解析

### 建议

先做治理方案，不直接切换。

## 推荐实施顺序

1. `Phase 7A`：定义 `novel-scan` 契约
2. `Phase 7B`：定义 selective-sync 方案
3. 只有两者约束足够稳定后，才进入真正实现

## 版本边界建议

- **`v0.10.0`**：如果完成契约与治理设计，并完成 docs/plan 收口
- 真正的实现版不建议和契约版混在一个 release 里

## 本阶段明确不做

- 不把 `novel-scan` 直接做成默认可用联网 skill
- 不直接切换 shared 资产同步策略
- 不在没有实验/验证前碰 `v1.0.0`

## 当前推荐下一步

第七阶段正式实施时，先从 **`novel-scan` 契约设计** 开始：

1. 明确数据源
2. 明确可信度分层
3. 明确输出格式
4. 明确“仅报告 / 可回写”的边界
