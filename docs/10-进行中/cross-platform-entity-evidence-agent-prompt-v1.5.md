# Cross-Platform Entity Evidence Agent Prompt (`v1.5`)

> Status: `supporting-prompt`
>
> 这是一份辅助执行提示词，依赖当前试行标准与 evidence pack；在标准仍 active 时，不建议单独迁移归档。

## 用途

这是一份可直接发给其他 agent 的执行提示词。

目标不是“先把 URL 填满”，而是按统一标准，把 profile 交叉压证据包补成**可复查的官方实体页证据**。

---

## 可直接转发的提示词

```md
你现在要为 Genm-codex 项目补 profile 交叉压证据包。不要脑补，不要用弱相关页面充数，只接受可复查的官方来源。

先读以下文档，再开始执行：

1. docs/10-进行中/cross-platform-entity-evidence-standard-v1.5.md
2. docs/10-进行中/batch-cross-platform-evidence-pack-v1.5.md

你的任务：

1. 从 evidence pack 中挑选一小批尚未完成的 profile
2. 把这些 profile 的平台入口从搜索页线索，替换为官方实体页证据
3. 按标准写回文档
4. 完成后运行：
   - bash scripts/validate-migration.sh
   - python3 -m pytest -q tests/test_profile_contract.py
5. 更新 progress.md，记录本轮完成的 profile、剩余缺口、验证结果

硬性标准：

- 只接受官方域名
- `A` 级：官方作品实体页
- `B` 级：官方分类 / 频道 / 榜单 / 专题页
- `C` 级：官方搜索页，只能做线索，不能作为最终完成项

单个 profile 的完成门槛：

- 番茄至少 1 条 `A`
- 起点至少 1 条 `A` 或 `B`
- 晋江至少 1 条 `A` 或 `B`
- 起点 / 晋江中至少一个平台必须是 `A`
- 最终条目中不得保留 `C` 级搜索页充当完成项

推荐目标：

- 最优：`A / A / A`
- 可接受：`A / A / B` 或 `A / B / A`

不可接受：

- `A / C / C`
- `B / B / C`
- 标题对不上、题材不匹配、页面不是官方域名

写回格式必须严格遵守标准文档中的模板：

### `<slug>`

- 状态：`complete | partial`
- 完成等级：`A/A/A`、`A/A/B` 等

### 番茄入口
- `A | <url>`
  - `<页面标题>`
  - 匹配：`<关键词1>` / `<关键词2>` / `<关键词3>`

### 起点入口
- `A | <url>` 或 `B | <url>`
  - `<页面标题>`
  - 匹配：`<关键词1>` / `<关键词2>` / `<关键词3>`

### 晋江入口
- `A | <url>` 或 `B | <url>`
  - `<页面标题>`
  - 匹配：`<关键词1>` / `<关键词2>` / `<关键词3>`

### 备注
- 如有必要，写平台偏差或证据不足说明

搜索策略优先级：

1. 优先用 Exa 搜官方域名，直接找 `A` 级实体页
2. 如果 `A` 级难找，再退到 `B` 级官方聚合页
3. 只在检索阶段临时使用 `C` 级搜索页定位，不要把它写进最终完成项

非常重要：

- 如果 `fetch` 失败，不要停，不要说“拿不到”，也不要把搜索页硬写进去
- 你必须主动换路继续找有效信息，例如：
  - 改用 Exa 搜官方域名结果
  - 改关键词，先搜题材名，再搜强标签，再搜 bucket
  - 先找作品页，再找分类页 / 榜单页 / 专题页
  - 同平台换另一条高置信官方实体页，不要困死在单个 URL 上
- 只有在尝试了多种官方检索方式后仍拿不到 `A/B` 级证据时，才能把该平台标记为缺口，并把该 profile 记为 `partial`

禁止事项：

- 不要脑补标题、标签、题材匹配理由
- 不要使用非官方站点、镜像站、采集站、论坛贴、聚合站
- 不要把 `search` 页、关键词页截图感知、模糊印象当作完成
- 不要因为某个平台难抓就跳过验证

工作方式：

- 每次只补 3-6 个 profile，优先补高把握题材
- 先做出 `A/A/A` 或 `A/A/B` 的小批样板，再继续扩
- 如果遇到长尾题材卡住，不要拖整批；把它标记为 `partial`，继续推进其他高把握对象

完成汇报必须包括：

- 本轮补了哪些 profile
- 每个 profile 的完成等级
- 哪些平台仍有缺口
- 运行了哪些验证命令，以及结果
```

## 使用建议

建议你给其他 agent 分批时，按下面方式发：

### 小批次分配

- 一次 3 到 6 个 profile
- 先选通用题材，再选长尾题材
- 每批都要求验证，不要攒到最后

### 推荐先做对象

优先级较高、较适合作为扩展示范的：

- `workplace-romance`
- `urban-daily`
- `urban-brainhole`
- `palace-intrigue`
- `historical-brainhole`
- `realistic`

### 暂时后放对象

更容易卡在证据稀疏或平台偏差上的：

- `cthulhu`
- `rule-horror`
- `rule-mystery`
- `zhihu-short`
- `dark`
- `dark-theme`

## 给调度者的提醒

如果某个 agent 回来时：

- 还在用搜索页充数
- 没写页面标题
- 没写匹配理由
- 没跑验证

那就不算完成，直接让它按标准返工。

看这份提示词时，优先检查三件事：

1. 是不是官方实体页
2. 完成等级有没有达到门槛
3. 有没有验证结果
