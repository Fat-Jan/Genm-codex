# 推进链与残账设计

## 目标

把“推进”从单纯的事件堆叠，变成有账本变化的推进。

## 基本链条

### 1. 事件发生

先有看得见的动作、冲突、交换或选择。

### 2. 信息差变化

事件之后，至少有一方知道了新的东西，或者误判被放大。

### 3. 条件交换

有人得到、失去、让步、暴露、欠下、押上。

### 4. 代价落地

推进不该只剩“事情顺利发生了”。

要问：

- 谁付了代价
- 代价是短期还是长期
- 代价是资源、关系、名声还是安全

### 5. 残账留存

本章结尾最好还有账没结清：

- 关系债
- 信息债
- 风险债
- 权力债

## 常见失真

### 1. 只有事件，没有账本

事情发生了，但局势没变。

### 2. 只有胜利，没有代价

读者会觉得推进很快，但也会觉得白拿。

### 3. 只有冲突，没有后果

打完、吵完、斗完，下一章像什么都没发生。

## 对工作流的要求

### `novel-outline`

章纲应尽量写清：

- 本章主要推进哪条线
- 哪个信息差在变化
- 本章结束后留什么残账

### `novel-write`

正文应尽量把“账本变化”写在场上，而不是只在总结里说已经变了。

### `novel-review` / `novel-precheck`

应检查：

- 推进是否过顺
- 证据/同盟/胜利是否过于自动
- 章末是否留下下一步驱动
