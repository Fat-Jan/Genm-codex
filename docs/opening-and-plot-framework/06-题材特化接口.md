# 题材特化接口

## 目标

第一版先定义以后怎么扩，不在这里一次性铺满所有题材资料。

## 每个题材后续都应补的四类信息

### 1. 开篇钩子偏好

这个题材的读者最先想看到什么：

- 压迫
- 异常
- 高概念
- 关系撕裂
- 规则亮相

### 2. 推进密度偏好

这个题材通常接受什么节奏：

- 快冲突
- 慢铺垫
- 高频小兑现
- 低频重兑现

### 3. 高频故障

这个题材最容易写坏在哪：

- 玄幻容易设定先行
- 宫斗容易全靠权力捷径
- 现言容易只有情绪、没有实际换账
- 悬疑容易只藏信息、不做情绪承接

### 4. 预检重点

投稿前最该检查什么：

- 标题/简介 promise 和正文是否一致
- 前三章兑现是否够
- 关系线是否成立
- 规则线是否自洽

## 当前第一版的用法

- 如果项目还没有专项特化文档，就先按通用规则执行
- 如果后续某个题材反复暴露同类问题，再单独扩一层题材卡
- 题材卡应作为 overlay，而不是重写通用骨架

## 番茄第一批特化策略

对于番茄路线，当前已经先落了一层优先级判断：

- [番茄优先分类清单（2026-03）](./fanqie-priority-categories-2026-03.md)

第一批先做 `P0` overlay：

- `宫斗宅斗`
- `职场婚恋`
- `青春甜宠`
- `豪门总裁`
- `都市日常`
- `玄幻脑洞`
- `都市脑洞`
- `历史脑洞`

对应卡片入口：

- [番茄 P0 特化卡](./fanqie-p0-overlays/README.md)

默认策略是：

1. 先用通用骨架判“开篇目标 / 构件 / 层次 / 推进链”
2. 再用番茄 P0 overlay 收紧对应 bucket 的开篇和推进要求
3. 只有真实项目持续出现同类误判时，才继续往更细的子路数或 tagpack 扩
