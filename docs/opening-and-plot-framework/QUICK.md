# 开篇方法与剧情层次框架 QUICK 速查卡

## 目的

把 `开篇方法与剧情层次` 框架的可执行规则压成一张速查卡，降低每次 review 时重复读取 6 份文档的成本。

它不是新的规则源，而是挂在现有：

- `core-constraints`
- Fanqie bucket 规则
- 开篇方法与剧情层次框架

之上的一张**速查执行门**。

一句话：

- 开篇方法负责"把开篇写成真正的故事入口"
- 本卡负责抓"开篇有没有抓力、推进有没有账本"

## 自动触发范围

当满足任一条件时，这张卡应自动生效：

- 任何 `novel-review` 或 `novel-precheck` 任务
- 任何涉及开篇、章纲的 `novel-write` 任务
- 显式传入 `opening_plot_check = true`

## 使用位置

### 对 `novel-review`

作为**强制第一遍漏斗**使用。

顺序固定：

1. 先判开篇有没有抓力
2. 再判推进有没有账本
3. 最后判剧情层次是否清晰

如果第一层不过，先报结构失败，不要先谈文风、节奏词或修辞。

### 对 `novel-write`

作为**轻量预检卡**使用。

写作阶段不做完整审稿，但必须提前防：

- 开篇只有背景，没有故事入口
- 推进只有事件堆叠，没有账本变化
- 剧情层次混乱，主线不清

## 漏斗逻辑

### 第一层：开篇有抓力吗

开篇必须同时满足 5 条：

1. 主承诺可见（读者能看见爽点方向、冲突方向、主角将面对的局）
2. 冲突前置（尽快出现压迫、异常、错位、风险、选择至少一种）
3. 情绪入口成立（读者知道该替谁急、恨、好奇、想看反击）
4. 未解问题成立（至少留下一个可追问的问题）
5. 近期兑现路径存在（前三章内存在一次明确兑现）

如果缺任意一条，默认按 `红` 看。

#### 不算通过的情况

- 大段背景先行，故事后到
- 首章基本没有能推动继续阅读的未解问题
- 前三章只有承诺，没有实际换账
- 读者不知道该替谁着急
- 只有设定，没有人物情绪

### 第二层：推进有账本吗

第一层过了，再看推进是不是"真推进"，不是"事件堆叠"。

重点抓 4 件事：

1. 事件发生后，至少有一方知道新东西或误判被放大
2. 有人得到、失去、让步、暴露、欠下、押上
3. 推进不该只剩"事情顺利发生了"
4. 本章结尾最好还有账没结清（关系债/信息债/风险债/权力债）

#### 高频红灯

- 只有事件，没有账本变化
- 只有胜利，没有代价
- 只有冲突，没有后果
- 只有推进，没有残账
- 章末只剩"真正的风暴还在后面"

### 第三层：剧情层次清晰吗

第二层过了，再看剧情层次是不是"真清晰"，不是"混乱堆叠"。

重点抓 5 件事：

1. 主线是什么（全书最主要的冲突方向和兑现方向）
2. 支线怎么服务主线（短期独立推进但长期服务主线）
3. 哪些人物线会反过来改变主线
4. 哪些关系线不是装饰，而是真会换账
5. 哪些世界/规则线会改动权力秩序或系统限制

#### 高频红灯

- 主线不清，支线乱飞
- 人物线只影响角色自己，不改变主线
- 关系线只是装饰，没有换账
- 世界/规则线只是背景，没有成为阻力或助力

## 判级

- `绿`
  - 三层都过
  - 本章可继续
- `黄`
  - 第一层过
  - 但第二层或第三层明显偏弱
  - 优先补核心推进或层次场景
- `红`
  - 第一层不过
  - 或第二层严重不过
  - 本章本质是背景章 / 铺垫章，先重做核心抓力

## 根因模板

不要把"字数不够"当根因。

优先从以下五类里归因：

- `A 类：开篇无抓力`
- `B 类：推进无账本`
- `C 类：层次不清晰`
- `D 类：兑现缺失`
- `E 类：多项并发`

推荐的一句话写法：

- 这章不是单纯写短了，而是 `开篇无抓力 / 推进无账本 / 层次不清晰`，所以抓力、账本、层次一起塌了。

## `novel-review` 输出契约

当本卡生效时，review 结果应额外给出一段 `opening_plot_summary`：

```md
opening_plot_summary:
  funnel_grade:
  failed_layer:
  root_cause:
  main_promise:
  opening_grab:
  recent_payoff:
  chapter_layers:
```

要求：

- `failed_layer` 只能写第一层 / 第二层 / 第三层
- `root_cause` 用上面的 `A-E` 归类
- `main_promise` 一句话描述主承诺；若暂无可写 `none`
- `opening_grab` 一句话描述开篇抓力；若暂无可写 `none`
- `recent_payoff` 一句话描述近期兑现；若暂无可写 `none`
- `chapter_layers` 列出本章涉及的剧情层次（主线/支线/人物线/关系线/世界线）

## `novel-write` 轻量预检

写章前至少快速过这 5 问：

1. 这章的主承诺是什么（爽点方向、冲突方向、主角将面对的局）
2. 开篇有没有抓力（压迫、异常、错位、风险、选择至少一种）
3. 读者知道该替谁着急、恨、好奇、想看反击吗
4. 前三章内有明确兑现路径吗
5. 推进有账本变化吗（得到/失去/让步/暴露/欠下/押上）

如果前 3 问里有 2 问答不实，不要把本章直接写成正文，先回到章纲补开篇骨架。

## 快速修复动作

| 问题 | 最小修法 |
|------|----------|
| 背景先行 | 把解释压后，先让角色被一件事逼着动 |
| 只有压迫，没有交换 | 尽快补一次小反击、误导、换账或信息跃迁 |
| 承诺过大，近期兑现缺失 | 把第一次兑现做小，但必须做实 |
| 只有设定，没有人物情绪 | 补上读者情绪入口（替谁急、恨、好奇） |
| 只有事件，没有账本 | 事件后必须改动至少一本账（关系/信息/资源/风险/权力） |
| 只有胜利，没有代价 | 给成功补一个残账、代价或新麻烦 |
| 章末太满 | 把"全成了"改成"成了一半，但代价暴露了" |
| 层次混乱 | 明确本章主推进线是什么，次级线是什么 |

## 何时该走 `novel-fix`

- 问题集中在 1-2 个场景
- 开篇抓力能通过补冲突、补情绪入口修正
- 推进能通过补账本变化修正
- 不需要推翻章节目标或顺序

## 何时该走 `novel-rewrite`

- 开篇完全没有抓力，只是背景介绍
- 推进只有事件堆叠，没有账本变化
- 剧情层次混乱，主线不清
- 需要重建章功能和场景次序

## 开篇构件速查

### 六个高频构件

1. **冲突**：外部压迫、关系失衡、身份错位、规则卡死、资源争夺
2. **悬念**：让读者往下翻不是因为被讲解，而是因为还不知道答案
3. **情绪代入**：怒、急、惜、爽前压抑、危机式好奇
4. **信息差**：让读者或角色知道得不一样，从而推动误判、等待或恐惧
5. **主承诺**：复仇、逆袭、翻案、升级、情感反转、权力换账
6. **章末留钩**：本章必须留下至少一个未结清账目

### 五种常见故障

1. **背景先行**：先讲世界、设定、家谱、系统说明，真正发生的事来得太晚
2. **只有压迫，没有交换**：主角一直惨，但前几章没有任何局势变化
3. **承诺过大，近期兑现缺失**：简介和首章承诺很大，但前三章都没有第一轮兑现
4. **只有设定，没有人物情绪**：能看懂局面，但读者不知道该替谁急、恨、等
5. **章末无钩**：章末没有留下任何未结清账目，读者没有继续翻的驱动

### 修正优先级

1. 先修"有没有故事"
2. 再修"有没有交换"
3. 再修"有没有情绪入口"
4. 最后修语言细节
