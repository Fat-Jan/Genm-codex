# 写作基本功与内容标准框架 QUICK 速查卡

## 目的

把 `写作基本功与内容标准` 框架的可执行规则压成一张速查卡，降低每次 review 时重复读取 8 份文档的成本。

它不是新的规则源，而是挂在现有：

- `core-constraints`
- Fanqie bucket 规则
- 写作基本功与内容标准框架

之上的一张**速查执行门**。

一句话：

- 写作基本功负责"把正文写得可读"
- 本卡负责抓"正文执行是不是把大纲和规则写在页面上了"

## 自动触发范围

当满足任一条件时，这张卡应自动生效：

- 任何 `novel-review` 或 `novel-precheck` 任务
- 任何涉及正文执行的 `novel-write` 任务
- 显式传入 `writing_core_check = true`

## 使用位置

### 对 `novel-review`

作为**强制第一遍漏斗**使用。

顺序固定：

1. 先判正文是否可读
2. 再判内容是否达标
3. 最后判是否可投稿

如果第一层不过，先报执行失败，不要先谈结构或规则。

### 对 `novel-write`

作为**轻量预检卡**使用。

写作阶段不做完整审稿，但必须提前防：

- 场景落点不明确
- 叙述视角不稳定
- 信息投放过量
- 对白只在解释
- 动作不能承接关系与冲突
- 情绪只停在标签
- 段落节奏滑向提纲化

## 漏斗逻辑

### 第一层：正文可读吗

正文必须同时满足 7 条（v1 必查项）：

1. 场景落点明确（读者能快速看懂此刻发生了什么）
2. 叙述视角稳定（单个段落里不要随意切换"谁知道/谁在判断/谁在解释"）
3. 信息投放不过量（信息投放要服务当前冲突：当前误判/选择/交换/后果）
4. 对白不只在解释（对白必须带来变化：改变信息差/关系温度/站位/后续动作）
5. 动作能承接关系与冲突（关键对白前后要有动作锚点，体现压力/犹豫/试探/控制）
6. 情绪不只停在标签（通过后果和反应落地：声调变化/动作变形/决策转向/关系后果）
7. 段落节奏不滑向提纲化（避免长段解释压住动作、对白堆叠但没有动作切面）

如果缺任意一条，默认按 `红` 看。

#### 不算通过的情况

- 场景关系不明确，谁在对谁做什么不明确
- 章内主要推进不明确
- 开头被背景说明吞掉
- 中段滑入总结腔
- 章末没有真实后续驱动
- 信息差没有变化
- 关系没有变化
- 代价或残账没有变化

### 第二层：内容达标吗

第一层过了，再看内容是不是"真达标"，不是"勉强能读"。

重点抓 4 件事（四项内容标准）：

1. **看得懂**：场景关系明确、谁在对谁做什么明确、章内主要推进明确
2. **看得下去**：开头不被背景说明吞掉、中段不滑入总结腔、章末有真实后续驱动
3. **看得到变化**：信息差有变化、关系有变化、代价或残账有变化
4. **承诺接得住**：当前正文能支撑包装 promise、前三章能支撑开篇 promise

#### 高频红灯

- 背景先行：读者先收到设定说明，后收到故事入口
- 悬浮对白：角色一直在说，但没有动作、压迫、试探和后果
- 提纲代替演出：用概述交代本该在场上完成的冲突或交换
- 白拿推进：关键证据、同盟、资源、认可来得太省
- 假紧张：制造了紧张感，但没有对应的风险或代价
- 过度承诺：包装 promise 明显大于正文近期承载

### 第三层：可投稿吗

第二层过了，再看是不是"真可投"，不是"勉强能看"。

重点抓 3 件事（三档投稿判断）：

1. **可继续追读**：开头 promise 清楚、当前章有真实变化、章末有后续驱动
2. **可投稿**：除了可追读，还应满足开篇与黄金三章没有明显失格、当前包装没有明显过度承诺、review 结果没有留下结构级红灯
3. **应先收口再投**：包装 promise 明显大于正文近期承载、开头抓力主要靠标题不是靠正文、前三章仍以解释铺垫概述为主、review 已指出结构级失败仍未修

#### 高频红灯

- 包装 promise 明显大于正文近期承载
- 开头抓力主要靠标题，不是靠正文
- 前三章仍以解释、铺垫、概述为主
- review 已指出结构级失败仍未修

## 判级

- `绿`
  - 三层都过
  - 本章可继续/可投稿
- `黄`
  - 第一层过
  - 但第二层或第三层明显偏弱
  - 优先补核心执行或内容场景
- `红`
  - 第一层不过
  - 或第二层严重不过
  - 本章本质是执行失败，先重做核心段落

## 根因模板

不要把"字数不够"当根因。

优先从以下五类里归因：

- `A 类：执行不可读`
- `B 类：内容不达标`
- `C 类：不可投稿`
- `D 类：承诺不接得住`
- `E 类：多项并发`

推荐的一句话写法：

- 这章不是单纯写短了，而是 `执行不可读 / 内容不达标 / 不可投稿`，所以可读性、内容、投稿性一起塌了。

## `novel-review` 输出契约

当本卡生效时，review 结果应额外给出一段 `writing_core_summary`：

```md
writing_core_summary:
  funnel_grade:
  failed_layer:
  root_cause:
  readability_score:
  content_score:
  submission_readiness:
  critical_gaps:
  suggested_fixes:
```

要求：

- `failed_layer` 只能写第一层 / 第二层 / 第三层
- `root_cause` 用上面的 `A-E` 归类
- `readability_score` 1-10 分，评估正文可读性
- `content_score` 1-10 分，评估内容达标度
- `submission_readiness` 绿/黄/红，评估投稿就绪度
- `critical_gaps` 列出最关键的 1-3 个缺口
- `suggested_fixes` 列出最关键的 1-3 个修复建议

## `novel-write` 轻量预检

写章前至少快速过这 5 问：

1. 场景落点明确吗（读者能快速看懂此刻发生了什么吗）
2. 叙述视角稳定吗（单个段落里没有随意切换"谁知道/谁在判断/谁在解释"吗）
3. 信息投放服务当前冲突吗（不是过量背景说明吗）
4. 对白带来变化吗（不只是解释设定、复述上一章、代替作者总结吗）
5. 动作承接关系与冲突吗（关键对白前后有动作锚点吗）

如果前 3 问里有 2 问答不实，不要把本章直接写成正文，先回到章纲补执行骨架。

## 快速修复动作

| 问题 | 最小修法 |
|------|----------|
| 场景落点不明确 | 先让读者知道：人在哪、正在做什么、谁在对谁施压或交换 |
| 叙述视角不稳定 | 单个段落里锁定"谁知道/谁在判断/谁在解释" |
| 信息投放过量 | 信息投放必须服务当前冲突（误判/选择/交换/后果） |
| 对白只在解释 | 对白必须带来变化（信息差/关系温度/站位/后续动作） |
| 动作不能承接 | 关键对白前后要有动作锚点（压力/犹豫/试探/控制） |
| 情绪只停在标签 | 通过后果和反应落地（声调变化/动作变形/决策转向/关系后果） |
| 段落节奏提纲化 | 避免长段解释压住动作、对白堆叠但没有动作切面 |
| 背景先行 | 把解释压后，先让角色被一件事逼着动 |
| 悬浮对白 | 补上动作、压迫、试探和后果 |
| 提纲代替演出 | 用场景演出，不要用概述交代 |
| 白拿推进 | 关键证据、同盟、资源、认可必须有代价 |
| 假紧张 | 制造紧张感必须有对应的风险或代价 |
| 过度承诺 | 包装 promise 不能明显大于正文近期承载 |

## 何时该走 `novel-fix`

- 问题集中在 1-2 个段落
- 执行能通过补动作、补后果、补代价修正
- 内容能通过补变化、补驱动修正
- 不需要推翻章节目标或顺序

## 何时该走 `novel-rewrite`

- 执行完全不可读，只是文字堆砌
- 内容完全不达标，没有真实变化
- 不可投稿，包装 promise 远大于正文承载
- 需要重建章功能和场景次序

## 写作基本功速查

### 七项 v1 必查项

1. **场景落点是否明确**：读者能快速看懂此刻发生了什么
2. **叙述视角是否稳定**：单个段落里不要随意切换"谁知道/谁在判断/谁在解释"
3. **信息投放是否过量**：信息投放要服务当前冲突（误判/选择/交换/后果）
4. **对白是否只在解释**：对白必须带来变化（信息差/关系温度/站位/后续动作）
5. **动作是否能承接关系与冲突**：关键对白前后要有动作锚点（压力/犹豫/试探/控制）
6. **情绪是否只停在标签**：通过后果和反应落地（声调变化/动作变形/决策转向/关系后果）
7. **段落节奏是否滑向提纲化**：避免长段解释压住动作、对白堆叠但没有动作切面

### 六种常见失格

1. **背景先行**：读者先收到设定说明，后收到故事入口
2. **悬浮对白**：角色一直在说，但没有动作、压迫、试探和后果
3. **提纲代替演出**：用概述交代本该在场上完成的冲突或交换
4. **白拿推进**：关键证据、同盟、资源、认可来得太省
5. **假紧张**：制造了紧张感，但没有对应的风险或代价
6. **过度承诺**：包装 promise 明显大于正文近期承载

### 修正优先级

1. 先修"有没有执行"
2. 再修"有没有内容"
3. 再修"能不能投稿"
4. 最后修语言细节
