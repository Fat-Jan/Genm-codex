# Genm-codex 番茄优先执行计划

## 目的

这份文档把当前项目的主方向收窄成一句话：

- **先让默认工作流稳定地产出更适合番茄平台的内容，并优先围绕番茄真实内容池优化。**

不再以“同时吃多个平台”为当前主线。

---

## 当前判断

过去的多平台方向不是完全错误，但它在当前阶段会带来两个问题：

1. 平台约束互相打架
2. 工程重心从“能稳定产出内容赚钱”被拖成“什么都想兼顾”

所以现在最合理的路线是：

- **平台优先级单点收缩**
- **先把番茄做深**

当前执行口号定为：

- **明日资源，今日实现，效率至上**

---

## 北极星目标

不是“支持更多平台”，而是：

- 让项目默认工作流能稳定产出**更符合番茄平台主推方向**的内容
- 让包装层、正文、预检和学习回写，围绕番茄统一收敛
- 让真实小说项目有更高概率在番茄拿到有效数据和收益

---

## 为什么是番茄优先

### 1. 官方公开信号足够多

当前番茄官方公开可用信号包括：

- 官网榜单 / 分类榜
- 阅读榜 / 新书榜
- 作者专区公告
- 作者专区教程中心与图文分类说明
- 短故事功能与数据说明
- 活动与激励口径

这已经足够支撑第一轮“平台约束建模”。

但从现在开始，不应只看“榜单热度”，还要同时看：

- 热度
- 收益倾向
- 平台扶持
- 当前系统适配度

参考：

- [fanqie-paid-signal-matrix.md](/Users/arm/Desktop/vscode/Genm-codex/docs/fanqie-paid-signal-matrix.md)
- [fanqie-writer-zone-lessons.md](/Users/arm/Desktop/vscode/Genm-codex/docs/research/fanqie/fanqie-writer-zone-lessons.md)
- [fanqie-rule-priority-matrix.md](/Users/arm/Desktop/vscode/Genm-codex/docs/fanqie-rule-priority-matrix.md)

### 2. 当前项目已有番茄基础

现有系统已经有：

- `fanqie-style`
- `profile-tomato`
- `novel-precheck` 的番茄口径
- `novel-package` 的番茄包装判断
- `market_adjustments` 对番茄玄幻的项目级建议

也就是说，番茄并不是从零开始。

### 3. 写作收益目标更一致

当前阶段最重要的不是“理论覆盖面”，而是：

- 内容能不能落到更高概率有收益的平台内容桶

番茄优先更符合这个目标。

## 执行总顺序

### 第一步：先做番茄内容池研究

目标：

- 弄清番茄当前主推什么
- 不同内容桶各自需要什么

重点：

- 分类榜与新书榜
- 标题风格
- 简介风格
- 篇幅与节奏
- 短故事 vs 长篇的差异
- 保障计划 / 征文 / 激励所对应的收益导向分类
- 付费偏好互补信号（如红烛小说）

产物建议：

- `docs/fanqie-content-buckets.md`
- `docs/fanqie-paid-signal-matrix.md`

### 第二步：再把内容桶约束建模

目标：

- 把“番茄当前热推方向”转成项目内可消费的约束

例如：

- 题材桶
- 篇幅桶
- 视角偏好
- 开头速度
- 爽点兑现周期
- 标题风格模板
- 简介结构模板

产物建议：

- `docs/fanqie-bucket-constraints.md`

### 第三步：接进默认工作流

顺序建议：

1. `novel-genre`
2. **番茄内容桶判断**
3. `novel-outline`
4. `novel-package`
5. `novel-write`
6. `novel-precheck`

这里的关键是：

- **内容桶判断在包装层之前**

### 第四步：再看是否扩到正文链

当包装、预检、题材判断都稳定后，再决定：

- 是否让 `write`
- 是否让 `learn`
- 是否让 `precheck`

更深地消费番茄内容桶约束。

---

## 对现有系统的直接影响

### 1. `novel-package`

后续不再只看：

- 题材
- 平台
- market adjustments

还要看：

- 当前番茄内容桶
- 该桶的标题 / 简介 / 开头风格

### 2. `novel-precheck`

后续不再只做“番茄一般口径”检查，而是要看：

- 你是不是在正确的番茄内容桶里写
- 包装是不是对这个桶说对了话

### 3. `novel-genre`

后续可能需要从：

- 题材判断

升级到：

- 题材 + 番茄内容桶判断

### 4. `novel-scan`

这条线如果继续保留实验能力，最应该服务的不是“全平台扫描”，而是：

- **番茄内容池与题材桶扫描**

---

## 当前最值得做的三件事

### 1. 番茄内容池研究

优先级：最高

因为这是后面所有约束建模的事实基础。

### 2. 番茄内容桶约束清单

优先级：高

因为没有它，包装层和正文层都只能吃“泛番茄口径”。

### 3. 包装层接桶

优先级：高

因为这是最容易最快看到收益的地方。

### 4. 女频付费线第一梯队 MVP

优先级：高

因为当前目标不是“先把所有男频做完”，而是：

- 先把最可能产生收益的方向做成可运行样板

当前推荐：

- 长篇：`宫斗宅斗`
- 短篇 / 中短：`现实情感`、`现言甜宠`
- 第一批执行顺序：
  1. `现实情感`
  2. `宫斗宅斗`
  3. `传统玄幻`

对应模板参考：

- [fanqie-mvp-bucket-templates.md](/Users/arm/Desktop/vscode/Genm-codex/docs/fanqie-mvp-bucket-templates.md)

## 当前已落地的第一批接入

### 已接入

- `novel-package`
- `novel-precheck`
- `novel-outline`

### 当前意义

这说明系统已经不是只“知道要做番茄优先”，而是已经开始把：

- `content_bucket`
- 桶级约束

接入到：

- 大纲判断
- 包装判断
- 投稿前判断

### 下一步建议

如果继续推进这一线，优先顺序应是：

1. 继续稳住 `package / precheck / outline` 的桶级判断
2. 再考虑让 `genre`
3. 最后才看是否让 `write` 深接桶级约束
4. 女频付费线不后置，直接进入第一梯队 MVP

---

## 当前不建议做的事

### 1. 再回去扩多平台主线

当前阶段不值。

### 2. 先把短故事系统做成一整套大工程

先做内容桶与约束层，比直接造大系统更稳。

### 3. 先把 `novel-scan` 正式化成全平台默认能力

当前更值钱的是：

- 缩成番茄优先

---

## 一句话结论

当前最合理的产品主线，不是“全平台网文生成器”，而是：

- **番茄优先的内容生成与包装优化系统。**

先研究番茄在推什么，再把你的题材与流派逻辑往这个内容池靠，再让包装层和正文链去消费这些约束。
