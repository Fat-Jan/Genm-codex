# `v1.5` Registry Consumer 一致性与 `novel-scan` 边界复检 2026-03-31

> 目的：对 `v1.5` 下一阶段中的剩余两条方向做一次统一收口：
>
> 1. registry / consumer 读取面是否已经形成一致边界
> 2. `novel-scan` 是否仍保持“可消费但不主链化”的约束
>
> 范围：`profile-calibration-and-bucket-mapping.md`、`bucket-overlay-inventory.md`、`scan-result-contract.md`、`novel-scan-usage-and-source-matrix-v1.5.md`、`novel-query` 等 consumer 说明。

---

## 一、复检结论

**思考强度：L4 审慎**

本轮结论：**通过。当前 registry / consumer 读取面整体一致，`novel-scan` 仍保持“可消费但不主链化”的边界。**

更具体地说：

1. `bucket / strong_tags / tagpacks / narrative_modes / tone_guardrails` 的职责划分仍然清楚
2. 当前 consumer 层已经知道这些 registry / overlay 数据是“输入约束层”，不是自由扩权入口
3. `novel-scan` 已经有 contract 与 consumer 去向，但仍被限制在：
   - 市场信号层
   - project-local soft guidance 层
   - gate / review 候选层
4. 当前没有证据表明：
   - `novel-scan` 被提升为默认工作流必经步骤
   - `novel-scan` 被用来直接改 canon
   - registry 层被 consumer 错读成多个同级主桶系统

因此，后续重点不是重做结构，而是继续维持这套边界，并在新增 consumer / sidecar 时复检有没有越权读取。

**思考强度：L3 深度**

结论：**是。**

证据：

- [profile-calibration-and-bucket-mapping.md](file:///Users/arm/Desktop/vscode/Genm-codex/docs/00-当前有效/profile-calibration-and-bucket-mapping.md)

### 2.3 当前 consumer 是否把 registry 层读成了正确的约束层
**思考强度：L3 深度**

结论：**是。**

证据：

- [profile-calibration-and-bucket-mapping.md](file:///Users/arm/Desktop/vscode/Genm-codex/docs/00-当前有效/profile-calibration-and-bucket-mapping.md)

- registry 层仍是“对创作消费有约束力的输入层”，而不是第二状态系统

### 3.2 `novel-scan` 产物是否仍然是 sidecar / candidate，而不是真值
**思考强度：L4 审慎**

结论：**是。**

- 覆盖 shared profile

判断：

- 没有看到 sidecar 权限上升为真值入口

## 四、当前风险与后续建议

**思考强度：L4 审慎**

当前没有发现需要立即修正实现的中高风险问题，但仍有 2 类后续要持续盯住的风险：

### 4.1 文档层历史描述与现状不完全同步
当前状态：

- [bucket-overlay-inventory.md](file:///Users/arm/Desktop/vscode/Genm-codex/docs/00-当前有效/bucket-overlay-inventory.md) 已完成现状化整理
- [bucket-profile-slug-mapping.md](file:///Users/arm/Desktop/vscode/Genm-codex/docs/00-当前有效/bucket-profile-slug-mapping.md) 已收口为 registry / runtime / 测试证据入口
- 当前这类风险已从“已有代表文档失真”收缩为“后续新增 inventory / 设计稿时的回归风险”

风险：
- 后续读者仍可能把新出现的历史“待实现”段落误当当前缺口
- 如果新增 inventory / 设计稿不显式指向 registry / runtime / tests，容易再次出现“文档语义领先于或落后于实现”的漂移

建议：
- 后续如继续做文档瘦身，优先处理新增的 inventory / 设计稿类文档
- 继续保持“现状优先、历史降级、真值入口前置”的写法，而不是重新引入散落的设计草稿

### 4.2 新增 consumer 未来可能越权读取 sidecar / overlay
风险：
- 新 consumer 把 market guidance / bucket overlay / tagpacks 当成 canon 或默认主链输入

建议：
- 后续若新增 consumer 或重写 `novel-query / package / outline`，应继续复检：
  - 是否保持 `bucket` 为单主桶
  - 是否继续把 `market-adjustments` 当 soft guidance
  - 是否继续把 `research-candidates` 仅作为 gate / review candidate

---

## 五、本轮结论摘要

**思考强度：L3 深度**

当前 `v1.5` 的 registry / consumer 一致性总体健康：bucket / tags / modes / guardrails 分工清楚，consumer 没有把 registry 层误用成第二真值系统；`novel-scan` 也仍严格保持“可消费但不主链化、可参考但不改 canon”的边界。下一步最重要的不是继续加结构，而是把这套边界继续稳定地守住。
