# Shared Asset Governance Unified Rules v1

> 创建时间：2026-03-31
>
> 本文档将 profiles / references / templates 三域的治理逻辑统一成可复用总框架。

---

## 一、治理目标

1. **统一治理术语**：避免不同域使用不同分类词
2. **统一判断标准**：建立跨域适用的证据分级体系
3. **统一决策规则**：明确保护/降级/归档/删除的准入/退出条件
4. **统一 exception 管理**：建立可接受的 exception 类型和处理流程

---

## 二、统一术语定义

### 2.1 资产状态分类

| 术语 | 定义 | 说明 |
|---|---|---|
| **稳定保留** | 结构完整 + 运行可投影 + 测试有兜底 + 证据链完整 + 无需 exception | 最高质量状态 |
| **观察保留** | 证据尚不充分，但风险较高暂时保留的项 | 需补证 |
| **保护降级** | 从 protected_local_paths 移出，但文件本身保留 | 文件仍存在 |
| **归档** | 移出保护名单 + 放入 docs/90-归档/ + 保留兼容说明 | 不再受治理保护 |
| **删除** | 从文件系统中移除 | 通常只用于确认无任何消费且无兼容价值的项 |
| **例外 / exception** | 结构不完整但有合理原因暂时保留的项 | 必须有 exception_type 说明 |

### 2.2 证据分级体系

| 证据级别 | 定义 | 治理动作倾向 |
|---|---|---|
| **Level A：硬消费** | 被 scripts 直接读取；被主流程真实装载；被 tests 直接依赖；缺失会导致流程失败 | 继续保护 |
| **Level B：强软消费** | 被多个 skill/doc/workflow 持续引用；与主流程强绑定；虽非直接读取但实质影响输出 | 保护或观察保留，需补强证据 |
| **Level C：弱软消费** | 仅少量文档/单点 skill 引用；运行影响不稳定；缺失不一定立即暴露 | 倾向降级或观察保留 |
| **Level D：声明消费** | 仅在文档、清单、说明中声明存在；没有 scripts/tests/运行链证据 | 移出保护名单，必要时保留文件但降级治理级别 |
| **Level E：历史残留** | 已有替代机制；当前链路无直接使用；仅出于兼容或遗留保留 | 归档候选或删除候选 |

---

## 三、证据采集优先级

判断一个资产的消费强度，按以下优先级采集证据：

1. **scripts 直接读取**：最高优先级，shell/python 脚本直接读取文件
2. **主流程真实装载**：被 workflow 核心步骤使用
3. **tests 直接依赖**：测试用例明确断言该文件存在或内容
4. **skills 持续引用**：SKILL.md 中声明使用，且为多个 skill 引用
5. **docs/workflow 引用**：文档或工作流中提及，但非直接读取
6. **清单/表格声明**：仅在名单、表格中声明存在

**原则**：真实消费优先于声明存在。文档写了、skill 提到了不等于真实运行依赖。

## 四、治理决策矩阵

### 4.1 稳定保留条件

同时满足以下**全部五个维度**：

| 维度 | 检查内容 | 判断标准 |
|---|---|---|
| 结构完整性 | 文件存在且格式有效 | schema/contract 文件格式正确 |
| 运行投影可读 | 能被 scripts/tests 正确读取 | 有明确的读取代码 |
| 测试兜底覆盖 | 有测试用例验证该资产 | 测试能正常执行 |
| 消费证据状态 | Level A 或 Level B 证据 | 证据链完整 |
| 无需 exception | 不依赖 exception 说明 | 无特殊豁免需求 |

### 4.2 保护收缩判断

| 证据级别 | 默认动作 | 特殊情况 |
|---|---|---|
| Level A | 保持 protected | 无 |
| Level B | 保持 protected + 标记观察 | 如无替代且风险高，保持 |
| Level C | 降级保护或观察保留 | 有前序确认保护理由的项，保持 protected |
| Level D | 移出 protected，文件保留 | 除非有明确保留理由 |
| Level E + 有替代 | 归档 | 无替代且风险低，可观察保留 |
| Level E + 无消费无兼容 | 删除候选 | 建议先归档而非直接删除 |

### 4.3 归档标准

满足以下任一条件即可归档：

1. **Level E + 有替代关系**：已有新的机制替代
2. **历史兼容意义**：代表旧版 workflow 但有参考价值
3. **已移出保护名单的旧文件**：不在 protected_local_paths 中且无消费证据

### 4.4 删除标准

满足以下**全部条件**才可删除：

1. **Level E**：无任何 scripts/tests/skills 消费证据
2. **无兼容价值**：不是任何流程的必要参考
3. **无归档必要**：不代表有价值的历史方案
4. **确认无隐藏消费**：搜索确认无任何引用

## 五、Exception 管理机制

### 5.1 可接受的 Exception 类型

| exception_type | 说明 | 处理方式 |
|---|---|---|
| platform-specific | 平台特定原因 | 单独追踪平台差异 |
| primary-source | 主要来源平台特殊 | 以该平台证据为主要判断依据 |
| legacy-compatible | 遗留兼容需要 | 标注兼容范围和期限 |
| pending-upgrade | 等待升级完成 | 跟踪升级进度 |

### 5.2 Exception 处理规则

1. **exception ≠ 无效**：带 exception 说明的对象，需单独评估例外原因
2. **必须记录 exception_type**：无 exception_type 的 exception 不被认可
3. **定期复审**：带 exception 的对象应定期重新评估

## 六、域适配的治理策略

### 6.1 Profiles 域

| 特征 | 说明 |
|---|---|
| 消费类型 | 运行时消费者明确（tests/.mighty/state.json） |
| 治理策略 | 建立稳定保留门槛清单（五维检查框架） |
| 特殊项 | zhihu-short（exception_type: zhihu-primary-source） |

### 6.2 References 域

| 特征 | 说明 |
|---|---|
| 消费类型 | 消费链最隐性（多为 skill 声明引用） |
| 治理策略 | 关注 declared-only 幻觉，区分软消费与硬消费 |
| 特殊项 | consumer-read-manifest（已降级为 Reference） |

### 6.3 Templates 域

| 特征 | 说明 |
|---|---|
| 消费类型 | 多为数据模板（被 scripts 读取） |
| 治理策略 | 以 scripts/tests 消费证据为主要判断依据 |
| 特殊项 | Genre 类（有前序确认保护理由） |

## 七、统一执行原则

1. **先盘点、后搜证、再分级、再决策、再落地、再验证、再报告**：不允许跳过步骤
2. **真实消费优先于声明存在**：docs/skills 引用 ≠ 硬消费
3. **不做大爆破**：禁止一轮性大规模清空式修改
4. **不确定时不删，先观察保留**：宁可留观察项，不能为了收缩而激进删除
5. **有替代关系时先归档，不直接删**
6. **每批次落地后必须跑验证**：验证通过才算当前批次完成
7. **所有执行必须留下清单、证据表、决策依据和结果记录**

---

## 八、默认执行规则

### 8.1 默认直接推进的事项

- 建立和更新盘点表、证据表、决策表
- 对 Level D/E 且无替代的 declared-only 资产提出降级建议
- 对 Level E 且有替代关系的历史资产提出归档建议
- 对 Level A 硬消费项保留保护
- 对文档/报告/阶段总结做同步更新
- 每批次后跑验证脚本和 sync report
- 更新 sync-governance.json 中的保护名单（低风险批次）
- 创建归档说明文件

### 8.2 默认暂停并升级的事项

- 要移出保护的对象属于核心 schema / 核心 contract / 主流程模板（Level A）
- 要进行 shared 目录结构重构
- 发现现有规则彼此矛盾
- 发现某项改动可能影响主流程输出
- 要引入 breaking change
- 要大规模重写 sync-governance.json 结构
- 遇到证据严重矛盾无法自行判断的情况
