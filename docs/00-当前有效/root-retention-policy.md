# 根目录保留原则

## 目的

这份文档回答一个问题：

> 哪些文件可以继续留在仓库根目录，哪些应该优先进入 `docs/` 分层目录？

目标不是追求“根目录绝对干净”，而是避免：

- 历史文档不断堆回根目录
- 入口、提醒、归档、研究材料混在一起
- 接手者无法判断某个根目录文件为什么还在这里

## 根目录允许保留的类型

### 1. 项目级主入口
- `README.md`
- `AGENTS.md`

### 2. 当前主线 roadmap
- 当前正在推进的主线 roadmap
- 紧邻当前主线、仍需要作为上游参考的主线 roadmap

示例：
- `v1.6-roadmap.md` → 当前主线
- `v1.5-roadmap.md` → 紧邻当前主线的上游参考
- `v1.4-roadmap.md` / `v1.3-roadmap.md` → 更早的历史主线，但仍作为上游参考保留高可见入口

### 3. 高可见提醒入口
- 只承担提醒或路由作用，不承担完整真值职责的文件

示例：
- `架构问题跟踪.md`

### 4. 高可见归档记录（受限保留）
- 保留在根目录是为了快速回溯，但必须明确标注 `archived-*` 状态

示例：
- `ISSUES.md`
- `task_plan_archive.md`

## 根目录不建议长期保留的类型

以下文档应优先分流到 `docs/`：

### 1. 历史型阶段汇报 / session note
- 单次 review
- 单次 session-progress
- 单次 completion report

→ 优先进入 `docs/90-归档/阶段/`

### 2. 研究草案 / 候选方案 / 实验材料
→ 优先进入 `docs/20-研究实验/`

### 3. 当前进行中的协议 / 标准 / sidecar / evidence body
→ 优先进入 `docs/10-进行中/`

### 4. 当前有效规则 / 合同 / 使用说明
→ 优先进入 `docs/00-当前有效/`

## 使用规则

如果一个文件留在根目录，最好至少满足下面之一：

1. 它是当前主线高可见入口
2. 它是项目级入口或规则总入口
3. 它只是提醒入口，但正式真源另有位置
4. 它虽已归档，但保留在根目录能明显降低回溯成本，且已补状态说明

否则，优先把它迁入 `docs/` 对应分层目录。
