# Genm-codex 当前待处理方案 v1

> 生成时间：2026-03-31
>
> 适用范围：常态治理收口阶段、低风险治理残留闭环、历史台账整理、下一阶段主线准备。
>
> 使用方式：按本文 phase 顺序执行。默认先完成 Phase 1，再依次推进 Phase 2、Phase 3、Phase 4，最后统一汇报。

> 历史说明（2026-04-02）：本文是 2026-03-31 形成的阶段性处理方案；其中涉及 `v1.5` Phase 4 的内容仅保留为当时的后续主线准备判断，当前主线以 `v1.6-roadmap.md` 为准。

---

## 一、总体判断

当前项目已经不处在需要重新发起大规模专项治理的阶段，而是处在：

1. 常态治理链待最终闭环
2. 低风险残留项待清理
3. 历史台账与 active 层待收口
4. v1.5 后续主线可继续推进，但不应与治理收口混做

因此，后续处理应以“先闭环、再清残留、再整台账、最后再推进下一阶段主线”为基本顺序。

|---|---|---|---|---|

| Phase 2 | P1 | 低风险治理残留闭环 | 低风险治理清理 | L3 |
| Phase 3 | P2 | 历史台账与 active 层收口 | 认知收口 / 结构清理 | L3 + L4 |

## 三、Phase 方案

## Phase 1：月度阶段收口正式落地

### 目标
把常态治理链真正闭环，补齐“每周快照 → 双周复核 → 月度阶段收口 → 主 agent 复检”的最后一环。

### 需处理事项
1. 产出 2026-03 月度阶段收口正式报告
2. 复用现有模板填写：
   - 输入材料
   - 月初 / 月末指标对比
   - 本月治理动作
   - 异常闭环情况
   - 当前治理状态判断
   - 下月建议方向
   - 是否需要专项治理
3. 在正式报告中保留任务级思考强度标注

### 预期产出
- `docs/00-当前有效/monthly-governance-phase-close-2026-03.md`

### 排序原因
- 这是当前最明确且最重要的未完成项
- 周快照、双周复核、模板、复检规则均已就位
- 不补齐该报告，常态治理链仍然缺最后一环

### 风险
- 风险低
- 主要风险不是修改错误，而是如果不补，治理链逻辑上始终未完全收口

### 建议思考强度
- 指标与事实汇总：L2
- 异常闭环判断：L3
- 是否需要专项治理：L4

---

## Phase 2：低风险治理残留闭环

### 目标
把当前已明确存在、但还未彻底解释或清理的低风险治理残留做一轮轻量闭环。

### 需处理事项

#### 2.1 templates `source-only 6` 路径级拆解
- 跑 report 与路径核对
- 确认这 6 项分别对应哪些路径
- 判断其性质：
  - 正常 source-only
  - 待观察项
  - 需补说明项
  - 需后续专项项

#### 2.2 `progress.md` 中 orphan 历史路径残留
- 决定是否改写为“已归档 legacy 记录”
- 或保持现状但补充更清晰的状态说明

#### 2.3 weekly snapshot 中历史发现的残留记录
- 保留历史发现价值
- 如需文档瘦身，可统一措辞为“历史发现 / 后续已闭环”

### 预期产出
- 一轮低风险闭环记录
- 如有必要，更新：
  - `weekly-governance-snapshot-2026-03-31.md`
  - `biweekly-governance-review-2026-03-31.md`
  - `progress.md`

### 排序原因
- 这些项都不是 blocker
- 但它们是当前治理面上最明显的“未完全闭环点”
- 做完后，常态治理状态会更干净、更稳定

### 风险
- 低风险
- 需避免把报表信号过度解释成确定事实

### 建议思考强度
- 报表拆解：L3
- 文档残留整理：L2
- 是否升级问题级别：L4

---

## Phase 3：历史台账与 active 层收口

### 目标
把“已经解决但仍挂在 active / 待办语义里的文件”做一轮结构化清理，减少后续误判和重复确认。

### 需处理事项

#### 3.1 清理 `docs/10-进行中/` 中已不再真正 open 的内容
重点关注：
- `architecture-open-issues.md`
- active 层中实际已 resolved、但仍保留 active 语义的文件

#### 3.2 清理旧治理报告中的过期待办语义
重点关注：
- `本轮整体治理报告.md`
- `templates-域巡检报告.md`

目标是避免后续接手者继续误判以下事项仍未完成：
- G5 14 项最终处置
- outline 2 项确认
- Genre 9 项保护级别重新评估

#### 3.3 明确“历史待办”与“当前待办”的边界
- 统一文档口径
- 必要时增加状态说明或更新指针
- 如需要，可将已失效的台账语义迁入归档说明层

### 预期产出
- 对 active 层文档做状态清理或补说明
- 对旧治理报告进行状态修正
- 必要时补一份“历史待办清洗说明”

### 排序原因
- 这类问题不影响运行主流程
- 但会显著影响后续判断质量与协作成本
- 若不处理，后续容易重复讨论“这个到底还算不算没做完”

### 风险
- 中低风险
- 涉及历史记录与现状真相边界，处理时需保守

### 建议思考强度
- 文档梳理：L2
- 状态迁移与口径裁决：L4

## Phase 4：v1.5 后续主线推进准备

> 历史说明：本 Phase 4 仅保留为 2026-03-31 时对 `v1.5` 后续主线的准备判断；当前主线已切到 `v1.6`。

### 目标
在治理链收口后，再继续推进 v1.5 后续主线，而不是与治理收口混做。

### 需处理事项
1. 重新审视 `v1.5-roadmap.md`
2. 区分：
   - 已完成前置
   - 可继续推进主线
   - 仍应保持 deferred 的项
3. 明确下一轮主线优先方向，例如：
   - 消费层产品化
   - sample library / regression 治理
   - profile / bucket / tagpack registry 化
   - novel-scan contract 与 consumer 接线

### 预期产出
- 下一轮主线实施方案
- 或 roadmap 状态更新建议

### 排序原因
- 这不是当前缺口修复，而是下一阶段发展方向
- 应建立在前 3 个 phase 收口之后

### 风险
- 若现在直接推进，容易把“治理未完全收口”和“新主线扩面”混在一起

### 建议思考强度
- 方向判断与优先级裁决：L4

---

## 四、推荐执行顺序

按以下顺序推进：

1. Phase 1：先补月度正式收口
2. Phase 2：再清低风险治理残留
3. Phase 3：再清历史台账 / active 层语义
4. Phase 4：最后决定 v1.5 下一阶段主线

---

## 五、执行结论

如果只选一个最值得优先做的动作，那么应先执行：

> **Phase 1：月度阶段收口正式落地**

因为它最短、最确定、收益最大。完成后，项目会从“治理链基本齐”进入“治理链正式闭环”状态。
