**思考强度：L4 审慎**

- 当前未发现必须返工的已落地实现任务
- 主要交付物、consumer 接线和定向验证仍能对上仓库现状

因此，Phase 4 的目标不是重开大实施，而是：

1. 明确哪些方向已经完成，不应重复开工

3. 明确下一阶段真正值得推进的方向
4. 明确哪些事项继续保持 deferred，不进入当前主线

## 三、仍需持续复检的点

**思考强度：L3 深度**

- 它与 runtime boundary、repo truth 边界、creative intake 稳定性直接相关

建议：
- 将该项纳入后续常规复检清单，而不是单独重开专项

## 四、下一阶段最适合推进的方向

**思考强度：L4 审慎**

### 推荐方向 A：持续复检型推进
优先级：最高

内容：
- 继续以“验证链 + 入口一致性 + 文档边界”做轻量复检
- 避免在治理链刚收口时再同时打开多个产品化方向

适用原因：
- 当前仓库刚从专项治理切换到常态治理
- 最适合先稳定，而不是立刻继续大扩面

### 推荐方向 B：样本库 / 回归治理增强
优先级：中高

内容：
- 在既有 sample manifest / matrix 基础上，继续加强样本维护规则
- 让 sample library 更适合作为长期回归面，而不是一次性文档成果

适用原因：
- 该方向与当前 contract 已对齐
- 风险低于大范围 skill / runtime 方向变动

### 推荐方向 C：profile / bucket / tagpack registry 读面增强
优先级：中

内容：
- 在已有 registry 读面基础上，继续提升消费侧一致性
- 更偏“增强已有读面”，而不是继续扩资产规模

适用原因：
- 当前前置与 consumer 已就位
- 适合做局部增强

### 推荐方向 D：novel-scan consumer 使用面增强
优先级：中

内容：
- 强化 status / query / package 等既有 consumer 的使用说明与复检
- 保持“可消费但不主链化”的边界

适用原因：
- 当前 contract 已定
- 但 roadmap 明确不应把 scan 变成默认 workflow 必经步骤

   - 当前只允许 contract + consumer integration
   - 不建议进入默认 workflow 必经步骤

## 六、Phase 4 结论

**思考强度：L4 审慎**

Phase 4 的最终结论是：

1. `v1.5` 主体 roadmap 当前**无需返工**
2. 后续更适合做“持续复检 + 轻量增强”，而不是重开大实施
3. 最值得继续推进的主线是：
   - 先做持续复检型推进
   - 再做样本库 / 回归治理增强
   - 再按需要推进 registry 读面增强与 scan consumer 使用面增强
4. 明确继续保持 deferred 的边界：
   - skill 真合并
   - runtime/plugin/daemon 实装
   - scan 默认主链化
