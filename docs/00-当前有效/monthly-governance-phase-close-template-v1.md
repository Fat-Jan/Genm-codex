# Genm-codex 月度阶段收口模板 v1

> 目的：将每周快照、双周异常复核、本月治理动作统一收口为一份正式月度报告，形成可追踪、可复检、可交接给其他 agent 执行的常态治理闭环。
>
> 适用对象：主 agent、执行 agent、复核 agent。
>
> 使用方式：执行 agent 可直接按本文逐节填写并执行；主 agent 按本文末复检规则审查结果，无需再临时设计月度收口结构。

---

## 一、执行说明

本模板用于月度阶段收口，重点不是重新发起专项治理，而是：

1. 汇总本月已完成的治理动作
2. 对比月初 / 月末治理指标
3. 判断是否存在新增风险积累
4. 说明哪些问题已闭环、哪些仍保留观察
5. 给出下月是否需要专项治理的明确结论

执行时默认遵循以下原则：

- 先汇总事实，再解释原因，再给结论
- 报表项与已验证事项分开写，避免过度推断
- 低风险项可直接记录与修正，高风险项单列升级
- 文档结论必须能回指到验证结果、证据表或现有治理文档
- 月度报告优先级高于零散聊天记录

---

## 二、思考强度标注规则

为方便协作与复检，本模板要求每类任务标注思考强度。

| 强度标签 | 含义 | 适用场景 | 产出要求 |
|---|---|---|---|
| **L1 轻量** | 以整理、记录、格式化、直接核对为主，几乎不涉及推理分歧 | 汇总已知结果、抄录验证输出、更新指标表 | 结果准确、字段完整 |
| **L2 标准** | 需要做常规判断、交叉核对和轻度解释，但证据边界较清晰 | 对比月初月末指标、归类已完成动作、整理遗留项 | 有依据、有分类、有简短解释 |
| **L3 深度** | 需要多源证据综合、边界判断、风险定性，存在一定误判成本 | 判断异常是否闭环、识别治理漂移、提出下月重点方向 | 明确证据链、说明判断逻辑 |
| **L4 审慎** | 涉及高风险结论或可能影响后续治理方向，需要保守判断 | 判断是否重新发起专项治理、是否升级高风险项、是否调整治理策略 | 保守结论、列出条件与边界 |

默认要求：

- 事实汇总类任务：L1
- 一般归类与对比类任务：L2
- 风险判断与趋势分析类任务：L3
- 治理方向裁决类任务：L4

## 三、月度执行任务清单

执行 agent 应按以下顺序推进，并在交付中保留每项任务的思考强度标注。

### 任务 1：收集本月输入材料
**思考强度：L1 轻量**

必收集：

- 本月至少一份 weekly snapshot
- 本月至少一份 biweekly governance review
- 当前有效的阶段报告 / 治理规则文档
- 本月涉及的实际治理结果、决策表、证据表
- 最新验证输出：
  - `bash scripts/validate-migration.sh`
  - `bash scripts/sync-shared-from-genm.sh --report`

最小要求：

- 明确列出本轮使用的输入文档路径
- 不遗漏当前有效基线文档
- 验证输出需为本轮最新结果

---

**思考强度：L2 标准**

至少对比以下指标：

- profiles protected / local-only / drift
- references protected / local-only / drift
- templates protected / local-only / drift / source-only
- 本月新增归档数量
- 本月新增观察项数量
- 本月已闭环异常数量

- 指标来源清楚
- 变化项要有简短说明
- 对无变化项也要明确写出“稳定”或“无变化”

---

### 任务 3：汇总本月治理动作
**思考强度：L2 标准**

按以下类别汇总：

1. 保护收缩
2. 归档
3. 文档修正
4. 验证增强
5. 异常修复
6. 规则固化

建议表格：

| 动作类别 | 涉及文件/范围 | 已执行内容 | 验证方式 | 结果 |
|---|---|---|---|---|

要求：

- 不写模糊描述，尽量写到文件或范围
- 每项动作最好对应验证或证据来源
- 已完成与仅建议动作分开写

---

### 任务 4：判断本月异常闭环情况
**思考强度：L3 深度**

至少覆盖：

- 上月遗留项是否已闭环
- 本月新增异常是否已定性
- 是否存在“报表异常被过度解释”的情况
- 是否仍有 orphan / declared-only / 历史残留扩散迹象
- 是否存在保护名单无依据膨胀

建议表格：

| 异常项 | 来源 | 当前状态 | 证据 | 是否闭环 | 说明 |
|---|---|---|---|---|---|

判断要求：

- “已闭环”必须有明确证据
- “待观察”要说明为什么暂不升级
- “未闭环”要指出阻塞点
- 仅报表项未拆解时，不得伪装成已确认事实

---

### 任务 5：提炼当前月度治理结论
**思考强度：L3 深度**

至少回答以下问题：

1. 当前 shared 三域是否稳定
2. 本月主要收益来自哪些治理动作
3. 当前最大的未决风险是什么
4. 项目处于“稳定维护 / 轻度修正 / 需要专项治理”哪种状态

要求：

- 结论必须与前文指标、动作、异常表一致
- 不允许只给抽象判断，不给依据
- 若整体稳定，也要指出仍需关注的低风险信号

---

### 任务 6：判断下月是否需要专项治理或策略调整
**思考强度：L4 审慎**

必须明确给出以下结论之一：

- 不需要，继续按周 / 双周 / 月节奏推进
- 需要发起轻量专项
- 需要发起正式专项治理
- 需要主 agent 介入裁决

建议写法：

- 结论
- 触发原因
- 风险等级
- 建议动作
- 是否必须升级

判断规则：

- 只要验证链正常、指标稳定、异常可解释，默认不升级为专项
- 若出现验证失败、protected 异常波动、核心资产路径失效，则必须升级
- 对证据冲突项要保持保守，不得为了“月度收口完整”强行定性

## 五、可直接给 agent 的执行提示词

以下提示词可直接交给其他 agent 使用：

```md
请基于以下文档，执行一轮 Genm-codex 月度阶段收口：

- docs/00-当前有效/steady-governance-rhythm-v1.md
- docs/00-当前有效/governance-execution-review-plan.md
- 最近一轮 weekly-governance-snapshot
- 最近一轮 biweekly-governance-review
- 本月涉及的阶段报告 / 决策表 / 证据表 / 实际治理结果

执行要求：
1. 先收集输入材料，再整理月初/月末指标对比
2. 汇总本月治理动作，并区分“已执行”与“仅建议”
3. 对本月异常做闭环判断，不得把未拆解报表项伪装成已确认事实
4. 明确说明 shared 三域当前是否稳定
5. 明确判断下月是否需要专项治理
6. 输出正式 md 文档，建议命名为：
   - docs/00-当前有效/monthly-governance-phase-close-{{YYYY-MM}}.md

强制要求：
- 每个任务块标注思考强度：L1 / L2 / L3 / L4
- 所有结论都要能回指到输入文档、验证结果或证据表
- 若存在高风险或证据冲突项，必须单独列出，不得强行收口
- 低风险项可直接记录并给出保守处理建议

输出结构至少包括：
- 输入材料
- 执行摘要
- 指标对比
- 治理动作汇总
- 异常闭环情况
- 当前治理状态判断
- 下月建议方向
- 是否需要专项治理
```

## 六、主 agent 复检规则

主 agent 复检月度报告时，按以下标准判断。

### A. 通过
同时满足：

- 输入材料完整
- 指标对比与验证输出一致
- 治理动作可回指到真实变更
- 异常闭环判断没有过度推断
- 下月策略结论清楚且保守

### B. 基本通过
存在轻微问题，但不影响整体结论，例如：

- 个别说明不够精确
- 个别引用路径缺少补充说明
- 思考强度标注不够统一

### C. 部分通过
存在中等问题，需要补写或修正，例如：

- 指标表不完整
- 已执行动作与建议动作混写
- 异常项闭环证据不足
- 月度结论与前文证据不一致

### D. 不通过
存在原则性问题，例如：

- 伪造或跳过验证
- 关键指标无法追溯
- 把报表信号写成确定事实
- 对高风险项未升级却直接下结论

## 七、主 agent 复检清单

| 检查项 | 通过标准 | 结果 |
|---|---|---|
| 输入材料是否完整 | 已包含周快照、双周复核、现行规则与最新验证输出 |  |
| 指标表是否可追溯 | 月初/月末数值来源清楚 |  |
| 治理动作是否真实 | 能对应到文档、文件或验证结果 |  |
| 异常闭环判断是否保守 | 未把待确认项写成已确认项 |  |
| 是否明确区分已处理/待观察 | 有清晰边界 |  |
| 下月结论是否可执行 | 给出明确继续/专项/升级判断 |  |
| 思考强度标注是否合理 | 事实类偏 L1/L2，判断类偏 L3/L4 |  |
