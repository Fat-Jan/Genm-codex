# Genm-codex 常态治理节奏 v1

> 目的：将当前已经形成的 shared 资产治理成果，从“专项治理”转入“常态运行”。
>
> 适用对象：主 agent、协作 agent、后续执行者。
>
> 使用方式：按本文定义的周期节奏、默认动作、升级条件和交付物推进，无需每次重新设计治理流程。

---

## 一、背景

当前项目已经完成以下关键治理动作：

- profiles 域系统巡检
- references 域系统巡检
- templates 域系统巡检
- templates G5 14 项最终复核并改判为“非保护手工参考模板库”
- 三域统一治理规则文档
- 执行—复检闭环方案文档
- 阶段报告与多份治理清单

这意味着项目已经具备进入“常态治理阶段”的条件。

常态治理的目标，不再是发动大规模专项，而是：

1. 维持当前治理成果稳定
2. 尽早发现新增异常
3. 控制保护名单膨胀
4. 避免再次积累 orphan / declared-only / 历史残留
5. 让多 agent 协作可以按固定节奏推进

---

## 二、总体目标

常态治理节奏的总体目标是：

> 以低摩擦、低确认成本的方式，持续维护 shared 资产、治理规则、验证脚本与报告状态的一致性。

拆分为 4 个具体目标：

### 2.1 稳定目标
维持当前三域治理成果不回退：
- profiles：稳定保留门槛有效
- references：保护名单健康
- templates：保护项/归档项/参考模板库边界清晰

### 2.2 预警目标
在问题积累到需要专项治理前，尽早识别：
- 新 orphan 候选
- 新 declared-only 候选
- 保护项膨胀
- drift 异常增长
- 文档与文件状态不一致

### 2.3 控制目标
避免后续出现：
- 无依据新增保护项
- 已迁移文件长时间不归档
- 文档残留链接失效
- 参考型模板被误删

### 2.4 协同目标
让其他 agent 可以按固定节奏推进：
- 知道什么时候做什么
- 知道哪些能默认执行
- 知道哪些必须升级确认
- 知道最终要交什么给主 agent 复检

---

## 三、治理对象

常态治理主要覆盖以下范围：

- `shared/profiles/**`
- `shared/references/**`
- `shared/templates/**`
- `shared/validators/**`
- `shared/sync-governance.json`
- `scripts/**`
- `tests/**`
- `skills/**`
- `docs/**`

其中重点关注 3 类对象：

### A. 保护资产
即 `sync-governance.json` 中的 protected_local_paths。

### B. 非保护但保留资产
包括：
- 已降级文件
- 非保护手工参考模板库
- 观察保留项

### C. 历史兼容资产
包括：
- 已归档文件
- 已迁移但仍有文档痕迹的文件

---

## 四、节奏设计

常态治理采用三层节奏：

- **每周**：轻量快照
- **每双周**：异常复核
- **每月**：阶段收口

---

## 五、每周节奏（轻量快照）

### 5.1 目标
快速确认项目治理状态是否稳定，不做大动作。

### 5.2 必做动作
每周至少完成以下动作：

1. 运行：
   - `bash scripts/validate-migration.sh`
   - `bash scripts/sync-shared-from-genm.sh --report`

2. 记录关键快照：
   - profiles protected / local-only / drift
   - references protected / local-only / drift
   - templates protected / local-only / drift

3. 检查是否出现以下变化：
   - protected 数量异常变化
   - drift 明显增加
   - source-only 新增
   - 文档和文件状态不一致

4. 记录本周是否有：
   - 新增 shared 资产
   - 新增归档
   - 新增保护项
   - 新增观察项

### 5.3 交付物
- `weekly-governance-snapshot.md` 或阶段报告中的周更区块

建议字段：

| 维度 | 本周值 | 上周值 | 变化 | 说明 |
|---|---|---|---|---|
| profiles protected |  |  |  |  |
| references protected |  |  |  |  |
| templates protected |  |  |  |  |
| profiles drift |  |  |  |  |
| references drift |  |  |  |  |
| templates drift |  |  |  |  |

### 5.4 成功标准
- 快照完成
- 验证通过
- 没有未解释的异常变化

### 5.5 默认规则
每周快照只做记录与预警，不主动做大规模名单调整。

---

## 六、每双周节奏（异常复核）

### 6.1 目标
对两周内积累的异常项做一轮针对性判断，避免小问题拖成专项问题。

### 6.2 复核范围
重点复核以下类型：

1. **新增 declared-only 候选**
2. **新增 orphan 候选**
3. **新增 drift 异常项**
4. **新增观察保留项**
5. **归档后仍有残留引用的项**
6. **docs 提及但文件已删除/归档的项**
7. **新增 shared 文件但未说明消费链的项**

### 6.3 必做动作
1. 对双周内的新增异常做 grep / 证据复核
2. 将每项异常归入以下类别之一：
   - 保持现状
   - 补充说明
   - 降级保护
   - 归档
   - 删除候选
   - 升级确认
3. 对低风险项直接处理
4. 对高风险项汇总给主 agent

### 6.4 交付物
- `biweekly-governance-review.md`

建议字段：

| path | 问题类型 | 证据 | 建议动作 | 风险 | 是否已执行 |
|---|---|---|---|---|---|

### 6.5 成功标准
- 本双周新增异常都被定性
- 低风险项已消化
- 高风险项被单独标记

### 6.6 默认规则
- declared-only → 默认先降级候选
- 有替代关系的历史项 → 默认先归档候选
- 证据不足但风险高 → 默认观察保留
- 仅 docs 级参考消费 → 不视为硬消费，但不应贸然删除

---

## 七、每月节奏（阶段收口）

### 7.1 目标
形成一轮正式的阶段性治理总结，让项目保持可回溯、可复检。

### 7.2 必做动作
1. 汇总本月所有治理动作
2. 对比月初/月末的：
   - protected 数量
   - local-only 数量
   - drift 数量
   - 归档数量
   - 新增观察项数量
3. 说明本月处理了哪些：
   - 保护收缩
   - 归档
   - 文档修正
   - 验证增强
   - 异常修复
4. 更新一份正式阶段报告
5. 如有需要，提出下一轮专项入口

### 7.3 交付物
- `monthly-governance-phase-report.md`

建议包含：
- 执行摘要
- 关键指标变化
- 本月治理动作
- 遗留问题
- 下月建议方向

### 7.4 成功标准
- 本月有正式报告
- 所有治理动作可追踪
- 下月是否需要专项治理结论清楚

### 7.5 默认规则
月度阶段报告是正式收口文档，优先级高于零散聊天记录。

---

## 八、触发式治理（超出固定节奏）

除周/双周/月节奏外，以下情况触发即时治理：

### 8.1 立即触发条件
任一满足即触发：

1. `validate-migration.sh` 失败
2. sync report 出现大幅异常变化
3. protected_local_paths 被新增/删除但没有说明
4. 关键文件路径失效
5. 归档后主路径仍大量残留引用
6. 新增 schema/contract 文件但没有消费链说明
7. 核心域（profiles/references/templates）被大规模修改

### 8.2 触发后的动作
- 立即暂停常规节奏
- 建立一次专项异常处理清单
- 先验证风险，再决定是否需要专项治理

---

## 九、默认执行规则

为减少频繁确认，常态治理按以下规则执行：

### 9.1 可默认直接执行
以下事项无需单独确认：

- 运行验证脚本和 sync report
- 更新周快照、双周复核、月度报告
- 记录新增异常项
- 修正已确认的文档残留引用
- 对 declared-only 做降级建议
- 对明显历史项做归档建议
- 补充 evidence 表或变更说明

### 9.2 必须升级确认
以下事项必须升级给主 agent：

- 核心 schema / 核心 contract / 主流程模板要移出保护
- shared 目录结构重构
- `sync-governance.json` 模型重写
- 大规模保护名单重置
- 可能影响运行主流程的改动
- 证据严重矛盾、无法定性的问题

### 9.3 风险处理优先级
- 不确定 → 先保留
- 有替代关系 → 先归档
- 仅 docs/参考消费 → 保留但不升为保护
- 运行硬消费 → 继续保护

## 十、角色分工

### 10.1 主 agent
负责：
- 方向判断
- 高风险项裁决
- 最终复检
- 阶段性结论确认

### 10.2 执行 agent
负责：
- 周快照
- 双周异常复核
- 月度报告整理
- 低风险治理动作
- 文档同步修正

### 10.3 复核 agent（可选）
负责：
- 对 evidence 表、决策表做二次抽查
- 挑出误判项
- 控制误删与误归档

---

## 十一、最小交付物要求

任何一轮常态治理，无论大小，至少应保留以下信息：

1. 本轮时间范围
2. 本轮检查范围
3. 本轮发现的问题
4. 本轮实际动作
5. 验证结果
6. 是否存在待升级事项

如果缺这些信息，后续复检时视为治理记录不完整。

---

## 十二、与当前项目基线的衔接

当前基线下，常态治理应特别注意：

### 12.1 profiles
- 继续沿用 `profile-stability-threshold-checklist-v1.md`
- 不要轻易推翻已确立的 complete / exception 判定

### 12.2 references
- 当前 10 项保护项整体健康
- 不做无证据收缩

### 12.3 templates
- 当前 protected 为 37 项
- G4 的 5 项已归档
- G5 的 14 项已明确为“非保护手工参考模板库”
- 后续不应再次把 G5 直接当成删除候选，除非出现新证据

---

## 十三、当前建议执行顺序

在常态治理阶段，建议按以下顺序推进：

1. **先跑每周快照**
2. **再做双周异常复核**
3. **再出月度阶段报告**
4. **只有出现明显异常时才升级为专项治理**

也就是说，默认状态下：

> 不再主动发起大专项，而是通过固定节奏发现问题、消化问题、收口问题。

---

## 十四、成功判定标准

常态治理运行良好，至少应满足：

1. 周快照持续存在
2. 双周异常项不会无限积压
3. 月度报告能说明变化原因
4. shared 三域状态基本稳定
5. 没有无依据的保护名单膨胀
6. 没有长期未处理的 orphan / declared-only / 归档残留
7. 多个 agent 按此文档可以持续协作

---

## 十五、建议的下一步动作

常态治理文档落地后，最适合立即做的是：

1. 用本文件启动第一次**每周治理快照**
2. 把本文件和执行—复检闭环方案一起交给其他 agent
3. 让其他 agent 按本文节奏持续输出周报/双周复核/阶段报告
4. 主 agent 只处理高风险项和月度收口

---

> 核心原则：
>
> **让治理从“靠专项推进”变成“靠节奏维持”；让问题在变大之前被发现、被解释、被处理。**
