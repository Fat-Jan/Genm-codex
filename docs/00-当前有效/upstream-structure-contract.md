# 上游结构合同

## 目的

这份文档定义 `creative-brief / content-positioning / launch-stack / 总纲` 四层上游结构各自负责什么，避免同一类事实在多个 sidecar、多个 skill 文案里重复维护。

## 四层分工

### `creative-brief`

负责：

- 题材定位
- 核心卖点
- 读者动机 / 读者画像
- 预期风格
- 起盘阻力与代价

它是：

- 立项输入
- `novel-init / novel-outline / novel-package` 的上游入口

它不是：

- 完整 `总纲`
- 章节规划

### `content-positioning`

负责：

- 主 profile / 主 bucket
- `strong_tags`
- `narrative_modes`
- `tone_guardrails`
- 题材侧 package / write / review 压缩提示

它是：

- 题材定位 sidecar
- 组合题材与结构风味的压缩合同

它不是：

- 第二份 `总纲`
- 中后程剧情推进账本

### `launch-stack`

负责：

- 前 1-3 章起盘
- 第一次硬兑现
- 早期 retention 协议
- 起盘侧 package / precheck 风险提示

它是：

- 开篇期协议栈

它不是：

- 整卷结构
- 第 004 章以后的长期节奏合同

### `总纲`

负责：

- 整卷 / 整书节奏
- 主副线迁移
- 关键兑现
- 卷末钩子
- 伏笔与成长线
- 写前素材账本

它是：

- 整体结构真源

它不是：

- 起盘协议 sidecar
- 单纯包装口号集合

## 推荐读法

1. 先用 `creative-brief` 锁立项输入
2. 再用 `总纲` 锁整卷 / 整书结构
3. 对番茄前 1-3 章，再用 `launch-stack` 锁起盘协议
4. 如果项目进入组合题材 / 多主题卖点阶段，再用 `content-positioning` 输出压缩 consumer 提示

## 最小边界

- 不要把 `content-positioning` 写成第二份 `总纲`
- 不要让 `launch-stack` 承担中后程卷纲
- 不要跳过 `creative-brief` 直接在包装层发明项目核心卖点
