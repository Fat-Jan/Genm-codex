# Genm-codex `v1.2` Roadmap

## 目的

这份 roadmap 承接 `v1.1` 完成后的下一阶段工作。

它回答：

- `v1.1` 收口后，最值得继续推进的方向是什么
- 哪些是主线增强，哪些只是可选扩展
- 后续工作如何避免再次把过多内容塞回 `v1.1`

## 状态说明

- `[done]`：已完成
- `[in_progress]`：进行中
- `[planned]`：已确认要做
- `[blocked]`：受前置条件阻塞
- `[deferred]`：明确延后

## 联网标签

- `[local]`：本地即可完成，不依赖外部资料
- `[hybrid]`：结构部分本地可做，但内容值或判断需要外部校准
- `[online]`：核心判断明显依赖外部资料或平台现状

## 联网能力检查

已验证当前环境具备两层联网能力：

1. 当前会话可直接联网
2. 仓库内 `scripts/acquire_source_text.py` 可真实抓取番茄官方页面

本轮实测：

- `curl -I -L 'https://fanqienovel.com/writer/zone/tutorial?tab=1'` 返回 `HTTP/2 200`
- `python3 scripts/acquire_source_text.py 'https://fanqienovel.com/writer/zone/tutorial?tab=1' --pretty --min-body-chars 50`
  - 返回 `status = success`
  - `source_type = fetch_mcp`
  - `provider_trace.fetch.name = reader_proxy`

因此，下面的 `[hybrid]` / `[online]` 项可以按真实联网能力执行，而不是只停在设想层。

## 当前总判断

`v1.2` 不应该继续做“大量能力补线”。

`v1.1` 已经完成了：

- Profile Contract
- shared governance
- workflow tail closure
- content-positioning
- 当前有效文档真相层

因此 `v1.2` 更应该聚焦：

1. **扩大映射覆盖面**
2. **提升内容层质量**
3. **补真实样本 smoke**
4. **决定哪些实验能力值得继续正式化**

## 主线 D：实验能力去留判断

### D1. `novel-scan` 是否正式化

- Status: `[deferred]`
- Mode: `[online]`
- 目标：
  - 在已有 `v1.1` 边界基础上，重新判断：
    - 保持实验态
    - 还是进入下一阶段更稳定的候选链

### D2. `novel-log` 是否继续保留在默认主线

- Status: `[done]`
- Mode: `[local]`
- 目标：
  - 结合 trace log 的真实使用频率决定：
    - 保留默认主线
    - 还是降回专家能力
- 结论：
  - 当前建议降为**已接线的专家辅助能力**
- 依据：
  - trace log 现在已有真实写入方
  - 但默认恢复 / 诊断主链仍然是：
    - `novel-workflow`
    - `novel-resume`
    - `novel-snapshot`
  - `novel-log` 目前更适合做只读观察与排障辅助，而不是默认主线入口

## 暂不做

- 不重开多 profile merge runtime
- 不做多主桶并列引擎
- 不引入 monolithic runtime / plugin system
- 不把 MCP memory 升成项目真值层

## 推荐顺序

1. 主线 A：扩大映射覆盖
2. 主线 B：内容层质量升级
3. 主线 C：真实样本 smoke
4. 主线 D：实验能力去留判断
