## `v1.5` 的文档化补充说明

`v1.5` 的重点不是继续增加新的 skill 或新 runtime，而是把现有能力收成更稳定的消费面。对文档层来说，应把它理解为四条线并行推进：

1. 样本库与 regression 资产从“散文件 + 文档口径”转向 manifest / inventory 可消费。
2. profile / bucket / tagpack 从“规则散读”转向 registry / contract / overlay 边界清晰。
3. `novel-scan` 从实验技能转向“有 contract、可被 consumer 读取、但仍非默认主链必经步骤”的能力。
4. skill 体系只做 rationalization、alias、deprecation 准备，不在 `v1.5` 做破坏性大合并。

因此，`v1.5` 的文档层不应把重点写成“新增能力目录”，而应明确三类事实：

- 哪些 contract / ADR 已经定住。
- 哪些文档与 inventory 工作可由 `mimo` 立即推进。
- 哪些内容在 `v1.5` 明确只做准备，不做最终实现。

## 建议纳入正式入口的 `v1.5` 事实清单

后续无论写进 README、`start-here.md` 还是 `skill-usage.md`，都建议统一只暴露以下事实：

1. `v1.5` 当前关注 consumer / registry / sample governance / scan consumer。
2. `P1` 是 `mimo` 可立即推进的文档与 inventory 工作。
3. `P2` 只在对应 codex 前置产物实际完成后再进入。
4. `P3` 是 codex 的实现层任务。
5. `v1.5` 明确不做：
   - `31 -> 25` 破坏性真合并
   - 新 plugin/runtime/daemon 实装

这样可以把 roadmap 的核心边界暴露出来，但避免把入口文档写成任务管理器。
