# Genm-codex 第十一阶段范围设计

## 设计结论

前十阶段已经把项目从“迁移试验”推进到“有明确产品边界的技能系统”。

因此第十一阶段不再优先做：

- 新 skill 扩张
- 小范围零碎补丁

第十一阶段的核心任务切换为：

- **`v1.0.0` 边界定义**
- **默认工作流收敛**

---

## 为什么现在进入这一阶段

到 `v0.14.0` 为止，项目已经具备：

- 核心写作闭环
- 查询 / 状态 / 设定中台
- 编辑控制与回溯
- 学习与投稿前预检
- 包装层
- 市场信号进入项目级状态
- 质量链路由开始稳定化

继续无限堆能力的边际收益已经下降。

现在最值钱的问题变成：

1. 什么是默认工作流
2. 什么是默认推荐能力
3. 什么仍是实验能力
4. 达到什么条件才能进入 `v1.0.0`

---

## Phase 11A：`v1.0.0` 边界定义

### 目标

明确：

- 什么叫“默认稳定”
- 什么叫“实验能力”
- 什么是 `1.0.0` 的前置门槛

### 需要回答的问题

1. `novel-scan` 仍处实验态，是否阻止 `1.0.0`
2. packaging / quality routing 是否已经构成稳定默认工作流
3. `shared` full-copy 是否仍然可接受
4. 当前 `e2e-novel` 驱动验证是否足够

### 第一版产物

- `docs/00-当前有效/v1-boundary.md`

---

## Phase 11B：默认工作流收敛

### 目标

把当前真正推荐给用户的主线压缩成少数几条默认路径，而不是几十个 skill 平铺。

### 建议默认工作流

#### 1. 新书起盘

- `novel-init`
- `novel-genre`
- `novel-outline`
- `novel-package`

#### 2. 正文生产

- `novel-write`
- `novel-review`
- `novel-fix` / `novel-polish` / `novel-rewrite`

#### 3. 投稿准备

- `novel-precheck`
- `novel-package`
- `novel-export`

#### 4. 持续学习

- `novel-learn`
- `novel-status`

### 第一版产物

- 默认工作流文档
- README / start-here / skill-usage 的默认路径更新

---

## 不做什么

第十一阶段暂不做：

- 新的低价值 help/tutorial 命令
- shared profile 自动重写
- 大规模 state schema 重构
- 立即推进 `v1.0.0` 发布

---

## 版本边界建议

- **`v0.15.0`**
  - 如果 `v1.0.0` 边界和默认工作流文档落地

- **`v1.0.0`**
  - 仅当默认工作流、默认能力边界、实验能力边界都清晰且稳定时

---

## 当前推荐下一步

如果继续自动推进，第十一阶段最合理的第一项是：

1. 先写 `v1-boundary.md`
2. 再把默认工作流写成正式文档

这比继续补零散能力更值钱。
