## 结论

最佳首个实施落点不是继续改安装脚本，也不是先补文档，而是先落一份**机器可读的 host capability matrix 真值文件**。

原因：

- 当前仓库已经有多宿主安装、Trae rules、read-only MCP、alias helper 等基础资产。
- 现在真正缺的是把“哪些宿主支持什么、证据到哪一步、退化策略是什么”收敛成单一事实源。
- 如果先改 `install-skills.sh` 或继续手写文档，只会继续扩大脚本/文档漂移。

## 为什么是这个顺序

### 已有资产已经够多

仓库里已经存在：

- `scripts/install-skills.sh`
- `scripts/render_skill_alias_plan.py`
- `tests/test_skill_alias_plan.py`
- `.trae/rules/rule.md`
- `.trae/rules/project-context.md`
- `scripts/project_knowledge_mcp_server.py`
- `docs/00-当前有效/skill-usage.md`

这说明 v1.6 不是“从零做多平台安装”，而是要把现有能力**升维收口**。

### 仓库已有成熟实现骨架

这次优先复用的不是新发明，而是仓库现成的 contract 模式：

- `shared/templates/state-schema-v5.json`
- `shared/templates/workflow-state-v2.json`
- `shared/templates/workflow-state-v2.schema.json`
- `shared/templates/sidecar-freshness-registry-v1.json`
- `shared/templates/acquire-provider-registry-v1.json`
- `tests/test_state_contracts.py`

这套模式的共同点是：

1. 先有 repo-owned JSON truth
2. 再有 schema 锁结构
3. 再有测试锁合同
4. 脚本和文档都只消费真值，不各写一套事实

## 本轮已选择的第一落点

### 第一落点

新增：

- `shared/templates/host-capability-matrix-v1.json`
- `shared/templates/host-capability-matrix-v1.schema.json`
- `tests/test_state_contracts.py` 中对应 contract tests

### 这版先锁什么

第一版先锁以下事实：

- `host_id`
- `status`
- `verification_level`
- `skill_install_root`
- `install_mode`
- `supports_skill_discovery`
- `supports_alias`
- `supports_rules_context`
- `supports_hooks`
- `supports_mcp`
- `supports_native_invocation`
- `degrade_policy`
- `notes`
- `evidence`

### 这版为什么不先做 adapter

因为 adapter 是消费层；如果上游还没有稳定 truth file，adapter 只会继续把事实散落到 shell / python / docs 里。

所以顺序必须是：

```text
host matrix truth
  -> schema
  -> tests
  -> install / context / doctor adapter 消费
  -> 文档投影
```

## 这版 host 口径

当前先把进入范围的宿主都落进同一份 matrix：

- `claude`
- `codex`
- `opencode`
- `openclaw`
- `trae`

并显式区分：

- `baseline`
- `legacy`
- `experimental`

以及：

- `verified`
- `partial`
- `assumed`

这样后续不会再把“目标角色”误写成“已经验证完成”。

## 为什么不是先改 install-skills.sh

`scripts/install-skills.sh` 现在的定位应该是：

- **compatibility shim**

它仍然保留，但不应该继续充当宿主事实中心。

正确方向是：

- 宿主事实先写进 host matrix
- 安装脚本再从 matrix 消费安装根和安装模式

## 为什么不是先写 host-support 文档

因为文档天然是投影层。

如果没有 machine-readable truth source：

- 安装表会漂
- alias 说明会漂
- experimental / verified 边界会漂
- Trae / OpenCode / OpenCLAW 会被误承诺

所以文档应当在 matrix 落地后再投影。

## 第一版完成后，下一步怎么走

### Phase 2

把当前分散事实继续收敛到 manifest：

- canonical skill registry
- alias registry
- host capability matrix
- host support status

### Phase 3

再做薄 adapter：

- install adapter
- context adapter
- read-surface adapter
- doctor adapter

### Phase 4

最后补 doctor / self-check 与文档投影。

## 这次实施的最小验收标准

第一版只要求做到：

- host matrix 文件存在
- schema 存在
- matrix 能通过 schema
- 测试锁住当前 5 个 host 的 status / verification level
- 测试锁住安装根与 install mode
- 测试锁住 Trae 的 unsupported install 语义
- 测试要求 evidence 至少指向真实 repo 证据或明确 runtime observation

## 当前一句话判断

v1.6 的最佳解不是“继续做多平台安装补丁”，而是先把**跨宿主能力事实**收敛成一个可验证、可消费、可测试的上游真值层，然后再让安装、Trae 上下文、MCP 与文档都从它派生。
