# Genm-codex `v1.1` 路线草案

## 目的

这份文档回答：

- 如果 `v1.0.0` 之后要继续推进，最值得做的 `v1.1` 方向是什么

它不是发布承诺，而是当前最合理的增强路线草案。

---

## 路线判断

`v1.1` 不应该回到“大量扩新命令”的思路。

最合理的增强方向仍然围绕已经验证过价值的主线：

1. 包装层
2. 质量路由
3. 学习回写

---

## 候选主线

### 1. 包装层增强

优先级：高

目标：

- 让 `novel-package` 在真实项目里更稳定
- 减少包装与正文承载的偏差

候选改进：

- 包装层与已有 review / precheck 信号更强联动
- 更稳定的“主方案 / 备选 / 暂不推荐”结构
- 更清晰的包装更新条件

### 2. 质量路由增强

优先级：高

目标：

- 继续收紧 `review / fix / polish / rewrite / precheck` 的口径

候选改进：

- 降低 `review` 与 `precheck` 的口径差异
- 更稳定的 `recommended_next_action`
- 更明确的批量投稿判断

### 3. 学习回写增强

优先级：中高

目标：

- 让 `learned_patterns` 对后续正文写作的作用更实用

候选改进：

- 让 `write / polish / precheck` 更显式消费 learn 信号
- 减少 learn 结果的泛化和空泛描述

---

## 暂缓项

### `novel-scan` 正式化

继续暂缓，不建议作为 `v1.1` 主线。

原因：

- 它仍然是实验能力
- 当前不阻塞默认主线
- 它适合单独作为实验能力演进线

### shared 同步策略切换

继续暂缓。

原因：

- 工程价值高，但不直接提升默认写作主线
- 更适合在明确需要时再处理

---

## 当前推荐

如果后续继续推进 `v1.1`，最合理的优先顺序是：

1. 包装层增强
2. 质量路由增强
3. 学习回写增强

---

## 一句话结论

`v1.1` 最该做的，不是“更多命令”，而是：

- **把已经证明有价值的默认主线做得更稳、更一致。**
