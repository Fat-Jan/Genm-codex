# Genm-codex `v1.1` 路线草案

## 目的

这份文档回答：

- 如果 `v1.0.0` 之后要继续推进，最值得做的 `v1.1` 方向是什么

它不是发布承诺，而是当前最合理的增强路线草案。

---

## 路线判断

`v1.1` 不应该回到“大量扩新命令”的思路。

最合理的增强方向仍然围绕已经验证过价值的主线：

1. 包装层
2. 质量路由
3. 学习回写

并且其中最值得优先切入的新方向，不是“再加一个包装命令”，而是：

- **番茄短故事 / 中短故事适配**

它更像包装层之前的上游约束层。

---

## 候选主线

### 1. 包装层增强

优先级：高

目标：

- 让 `novel-package` 在真实项目里更稳定
- 减少包装与正文承载的偏差

候选改进：

- 包装层与已有 review / precheck 信号更强联动
- 更稳定的“主方案 / 备选 / 暂不推荐”结构
- 更清晰的包装更新条件
- 接入番茄短故事分类与内容桶约束，而不是只做长篇主站包装

#### 番茄短故事适配子线

建议顺序：

1. 先做短故事分类研究
2. 再做短故事约束建模
3. 再让包装层消费这些约束

而不是反过来。

参考：

- [fanqie-short-story-adaptation.md](/Users/arm/Desktop/vscode/Genm-codex/docs/fanqie-short-story-adaptation.md)
- [fanqie-content-buckets.md](/Users/arm/Desktop/vscode/Genm-codex/docs/fanqie-content-buckets.md)
- [fanqie-bucket-constraints.md](/Users/arm/Desktop/vscode/Genm-codex/docs/fanqie-bucket-constraints.md)

### 2. 质量路由增强

优先级：高

目标：

- 继续收紧 `review / fix / polish / rewrite / precheck` 的口径

候选改进：

- 降低 `review` 与 `precheck` 的口径差异
- 更稳定的 `recommended_next_action`
- 更明确的批量投稿判断

### 3. 学习回写增强

优先级：中高

目标：

- 让 `learned_patterns` 对后续正文写作的作用更实用

候选改进：

- 让 `write / polish / precheck` 更显式消费 learn 信号
- 减少 learn 结果的泛化和空泛描述

---

## 暂缓项

### `novel-scan` 正式化

继续暂缓，不建议作为 `v1.1` 主线。

原因：

- 它仍然是实验能力
- 当前不阻塞默认主线
- 它适合单独作为实验能力演进线

### shared 同步策略切换

继续暂缓。

原因：

- 工程价值高，但不直接提升默认写作主线
- 更适合在明确需要时再处理

---

## 当前推荐

如果后续继续推进 `v1.1`，最合理的优先顺序是：

1. **番茄优先内容池研究**
2. **番茄内容桶约束建模**
3. **包装层增强**
4. 质量路由增强
5. 学习回写增强

也就是说，真正顺序不是：

- 先做包装
- 再想平台

而应该是：

- 先做番茄内容池与分类桶
- 再让包装层消费这些约束

参考：

- [fanqie-first-execution-plan.md](/Users/arm/Desktop/vscode/Genm-codex/docs/fanqie-first-execution-plan.md)

---

## 一句话结论

`v1.1` 最该做的，不是“更多命令”，而是：

- **把已经证明有价值的默认主线做得更稳、更一致。**
