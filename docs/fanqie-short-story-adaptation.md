# 番茄短故事适配说明

## 目的

这份文档回答一个很实际的问题：

- 如果后续要做“番茄短故事 / 短篇 / 中短故事”的针对性优化，
- 这一层到底应该放在包装层前面还是后面。

结论很明确：

- **放在包装层前面。**

更准确地说：

- 这是一个**上游分类与平台约束层**
- 包装层是**下游表达层**

---

## 当前能确认的官方信号

以下信息来自番茄官方站点 / 作者专区公开页面：

### 1. 短故事的篇幅和风格有独立口径

官方公开口径至少出现过两种相近表述：

- 短故事篇幅在 `6000-80000` 字之间，`10000-30000` 字最佳
- 也出现过“`3万字以内`、一篇完结”的短故事说明

稳定共识是：

- 篇幅明显短于长篇连载
- 一篇完结
- 节奏更紧
- 钩子更重
- 常见第一人称或强代入视角

### 2. 短故事有独立的数据与工作台能力

官方页面明确提到：

- 作家专区 / 作者助手中存在短故事功能入口
- 短故事有独立的数据分析页面
- 可以查看作品数据、展示量、阅读量、收益等

这说明：

- 短故事不是“长篇模式的一个小变体”
- 它在平台侧就是单独管理、单独优化的对象

### 3. 番茄官网有官方榜单与分类榜单

番茄官网存在：

- 总榜 / 分类榜
- 阅读榜 / 新书榜

官方榜单口径显示：

- 按分类排榜
- 排序依据与在读数据相关
- 榜单每天更新

虽然这些榜单主要服务常规长篇主站内容，但它们仍能提供：

- 热门题材方向
- 标题风格信号
- 分类桶的竞争态势

### 4. 短故事还有单独活动/激励口径

番茄官方短故事活动里还出现过：

- 中短故事
- 深耕计划
- 锐笔计划
- 热点话题短故事挑战

这说明：

- 平台对短故事不是静态看待
- 它会按内容池、活动、投放渠道去做差异化运营

---

## 对系统设计的直接结论

### 1. 短故事优化不能只靠包装层

因为包装层只负责：

- 书名
- 简介
- 开篇表达

但短故事真正决定成败的，是更前面的东西：

- 你到底投哪个短故事内容桶
- 篇幅目标是多少
- 叙事视角是什么
- 开头几百字是冲突先行还是情绪先行
- 结尾要反转、反噬、情感回扣还是主题收束

这些都属于：

- **内容定位 / 分类约束层**

而不是包装层。

### 2. 正确顺序

正确顺序应是：

1. 先做番茄短故事分类研究
2. 再做目标桶判断
3. 再把分类约束写进项目侧规则
4. 再跑包装层

也就是说：

- **分类优化在前**
- **包装生成在后**

### 3. 如果以后做成系统能力

不建议直接把“短故事分类研究”塞进 `novel-package`。

更合理的是：

- 单独形成一层：
  - 平台内容桶 / 短故事约束层
- 然后让：
  - `novel-package`
  - `novel-outline`
  - `novel-precheck`
  去消费它

---

## 建议接入点

如果以当前默认工作流来放，最合理位置是：

1. `novel-init`
2. `novel-genre`
3. **短故事分类与平台桶判断**
4. `novel-outline`
5. `novel-package`

所以它不该放在包装层之后。

它应该作为：

- **包装层之前的上游约束**

---

## 对 `v1.1` 的启发

如果 `v1.1` 要做这条线，最合理的最小实现不是直接加“短故事写作大系统”，而是先做：

1. 番茄短故事适配说明
2. 短故事内容桶 / 分类约束清单
3. 包装层消费这些约束
4. precheck 对短故事口径单独判断

这会比直接盲目扩命令更稳。

---

## 当前推荐

如果你接下来真要做番茄短故事优化，最合理的路线是：

1. 先做分类研究
2. 再做约束建模
3. 再接包装层
4. 最后才看要不要进正文写作链

一句话：

- **番茄短故事适配是上游约束层，不是下游包装层。**
