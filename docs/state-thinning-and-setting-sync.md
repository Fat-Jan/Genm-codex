# State 瘦身与设定同步方案

## 目的

这份方案回答两个问题：

1. 为什么当前 `.mighty/state.json` 会越来越臃肿
2. 正确的长期设计，应该怎样把：
   - 运行态
   - 设定资产
   - 检索层
   - 同步层
   分开

目标不是推翻当前体系，而是：

- **保住当前可用性**
- **避免 `state.json` 继续膨胀**
- **补上设定集长期不更新的真实缺口**

---

## 一、当前问题是什么

当前体系默认把大量“当前可继续写作所需的信息”都塞进了 `.mighty/state.json`。

它现在同时承担了：

### 1. 运行态

- `progress`
- 当前主角状态
- 当前伏笔 / 悬念
- 当前主线进度

### 2. 派生索引

- `chapter_meta`
- `chapter_snapshots`
- `summaries_index`

### 3. 工作与策略信息

- `quality_metrics`
- `learned_patterns`
- `market_adjustments`

### 4. 部分长期事实

- `entities.characters.*`
- `entities.items.*`
- `entities.locations.*`

这会带来 4 个问题：

1. 文件越来越大，读写成本越来越高
2. “当前事实”和“长期设定”混在一起
3. `设定集/` 反而失去存在感
4. 长期项目里，人会越来越依赖 `state`，而不是稳定资产

---

## 二、正确设计的核心结论

**`state` 不该继续做“大而全总表”。**

正确方向应该是：

### A. `state.json`
做成：

- **运行态**

### B. `设定集/`
做成：

- **长期资产层**

### C. `.mighty/index.json`
做成：

- **检索层**

### D. `.mighty/market-data.json`
做成：

- **内容数据层 / 市场层**

### E. 新增同步层
做成：

- **把运行态里已经稳定的实体，沉淀回设定资产**

一句话版本：

- **`state` 管当前**
- **`设定集` 管长期**
- **`index` 管检索**
- **`sync` 负责把当前沉淀成长期**

---

## 三、哪些东西该留在 `state`

### 必留

这些是写下一章时必须知道的，应该继续留在 `state`：

- `meta`
- `progress`
- `entities.characters.protagonist`
- 当前活跃 `foreshadowing`
- 当前活跃 `suspense`
- `main_quest.progress`
- 最近若干章的 `chapter_meta`
- 最近若干章的 `chapter_snapshots`
- 最近若干章的 `summaries_index`
- 少量 `quality_metrics`
- 少量 `market_adjustments`
- 少量 `learned_patterns`

### 说明

这里的原则是：

- **只留“当前继续写作必须立刻读”的东西**

也就是：

- 下一章不读这些就容易接不上

---

## 四、哪些东西不该继续膨胀进 `state`

### 1. 全量角色资产

不该长期主要靠：

- `entities.characters.active`
- `character_states`

来扛。

更应该沉淀到：

- `设定集/角色/*.md`

### 2. 全量地点 / 势力 / 物品资产

不该长期主要靠：

- `entities.locations`
- `entities.items`

来扛。

更应该沉淀到：

- `设定集/地点/*.md`
- `设定集/势力/*.md`
- `设定集/物品/*.md`

### 3. 过长的 chapter history

`state` 里不应该无限长期保留所有章节的超细粒度历史快照。

建议：

- 只保留近段核心章节快照
- 更久的历史靠：
  - `index`
  - `chapters/`
  - `git`

### 4. 过重的学习与市场数据

如果后面：

- `learned_patterns`
- `market_adjustments`

继续膨胀，也不该继续塞在 `state` 里。

后续可以拆成：

- `.mighty/learned-patterns.json`
- `.mighty/market-adjustments.json`

当前先不强制拆，但方向应明确。

---

## 五、设定集应该承担什么角色

### `设定集/角色/`
负责：

- 主角卡
- 核心配角卡
- 重要反派卡
- 关系长期卡

### `设定集/地点/`
负责：

- 重要地点
- 重复使用地点
- 对情节有规则作用的地点

### `设定集/势力/`
负责：

- 组织
- 家族
- 宗门
- 宫廷势力

### `设定集/物品/`
负责：

- 反复使用的重要证据
- 关键物件
- 特殊道具 / 兵器 / 信物

也就是说：

- `设定集` 不是每章都要狂更
- 但它必须承担“长期事实沉淀层”的职责

---

## 六、同步层应该怎么做

### 同步原则

不是每一章写完就把所有新名词全写成卡。

而是只同步：

- **稳定出现**
- **后续还会继续用**
- **已经从正文里证明不是一次性路人**

的实体。

### 推荐同步范围

优先同步 3 类：

1. **活跃角色**
   - 连续出现
   - 对主线仍有作用
2. **重要地点**
   - 后续还要回到这里
   - 或者它本身有规则作用
3. **重要势力**
   - 已经明确对主线形成压制 / 支撑

### 推荐触发时机

不建议每章自动写。

建议这 3 种触发：

1. **章节里程碑触发**
   - 如每 3 章 / 每 5 章
2. **卷级触发**
   - 第一卷结束
   - 第二卷开启
3. **显式调用触发**
   - 用户或工作流明确要求同步

### 推荐实现方式

推荐后续新增一个显式能力，例如：

- `novel-sync`
或
- `setting-sync`

它读取：

- `.mighty/state.json`
- `chapter_meta`
- `summaries_index`
- 必要时最近章节正文

再决定：

- 哪些角色要补卡
- 哪些地点要补卡
- 哪些势力要补卡

---

## 七、同步时的真值顺序

为了避免“写出两份互相打架的事实”，同步层必须遵守这一套顺序：

1. **当前正文 + 当前 state**
2. **总纲 / 章纲**
3. **已有设定卡**

如果三者冲突：

- 优先信：
  - 当前正文
  - 当前 state
- 然后修：
  - 设定卡

也就是说：

- `设定集` 是沉淀层
- **不是比正文和 state 更高的神谕层**

---

## 八、与现有技能的关系

### `novel-write`

当前职责保持不变：

- 写正文
- 更新 `state`

不建议让它自动大规模更新设定集。

### `novel-character`

后续可增强为：

- 支持从 `state` / 最近章节中半自动补角色卡

### `novel-setting`

后续可增强为：

- 支持从 `state` / 最近章节中半自动补地点 / 势力卡

### `novel-retrieve`

后续应优先读：

1. `state`
2. `设定集`
3. `index`

并更明确提示：

- 当前是“运行态”
还是
- “长期资产描述”

### `novel-index`

应继续保持：

- 检索层

不升级成真值层。

---

## 九、最小可落地路线

### Phase 1：先定边界

先明确：

- 哪些字段继续留在 `state`
- 哪些以后不再鼓励膨胀进去

### Phase 2：先补同步能力

第一版只做：

- 活跃角色同步
- 重要地点同步
- 重要势力同步

当前已落地的第一版实现：

- `novel-sync`
- `scripts/sync-setting-assets.py`

### Phase 3：再考虑进一步瘦身

等同步层稳定后，再判断是否拆：

- `learned_patterns`
- `market_adjustments`

到独立文件。

---

## 十、最终结论

正确设计不是：

- 继续把所有东西塞进 `state`

也不是：

- 完全不用 `state`

而是：

- **让 `state` 回到运行态**
- **让 `设定集` 成为长期资产层**
- **让 `index` 专注检索**
- **新增同步层，把已稳定的正文事实沉淀成设定卡**

如果后续要实现，这份方案的第一优先级不是改写所有技能，

而是：

- **先做一个最小的 `setting-sync`。**
