# 反脸谱化工作流使用建议

日期：2026-03-22

## 目的

把两条真实 smoke 的经验压成可直接执行的路由建议，回答三个问题：

1. 什么时候只用 `write -> review` 就够了
2. 什么时候必须走 `review -> fix -> 再 review`
3. 什么时候该把问题上推回 `outline`

这份文档不新增规则，只把现有规则变成更好用的流程判断。

## 一句话总原则

**反脸谱化不是独立工作流，而是挂在默认写作主链上的结构门。**

也就是：

- `outline` 负责提前锁结构
- `write` 负责不把人写回按钮
- `review` 负责指出本章究竟是“还能继续写”还是“必须先修”
- `fix` 负责做局部结构修补
- `precheck` 负责把结构失衡当投稿前风险

## 最小使用方式

如果你只想最低成本地用这套体系，按这条走：

1. `novel-outline`
2. `novel-write`
3. `novel-review`

然后看路由：

- `recommended_next_action = novel-write`
  - 继续写下一章
- `recommended_next_action = novel-fix`
  - 先局部修，再继续
- `recommended_next_action = novel-rewrite`
  - 回退到结构层重做

## 三种典型路线

### 路线 A：直接通过型

适用特征：

- 关系裂口还活着
- 支持者不是自动站队
- 对立角色有自己的算盘
- 本章赢的是范围、位置、局部优势，不是“一口吃完整个答案”

工作流：

`outline -> write -> review -> write`

项目证据：

- [双女主替身交叉验证](./cross-genre-smoke-dual-substitute-evil-2026-03-22.md)

对应结论：

- `review_score = 88`
- `recommended_next_action = novel-write`

这类章节通常不需要再修，只需要在下一章继续维持：

- 裂口
- 多方算盘
- 代价链

### 路线 B：局部修补型

适用特征：

- 核心结构是对的
- 但主角信息命中率偏虚
- 或支持者开始往单声道滑
- 或高潮阻力偏弱，收益太顺
- 或正文长度已经挤掉了结构张力

工作流：

`outline -> write -> review -> fix -> review -> precheck`

项目证据：

- [宫斗宅斗真实链路](./real-chain-smoke-e2e-gongdou-evil-2026-03-22.md)

对应结论：

- 首轮：`review_score = 82`，`recommended_next_action = novel-fix`
- 二轮：`review_score = 88`，`recommended_next_action = novel-write`

这类章节最适合 `novel-fix` 的问题簇：

- 信息来源略虚
- 对立面反制不够
- 关系 / 阵营动作还不够尖
- 章内说明过多、压垮倒计时

### 路线 C：上推重构型

适用特征：

- 人物动机和事件顺序根本不兼容
- 支持者和对立方都只剩功能位
- 冲突靠误会和作者强推，不靠条件与代价
- 主角一人吞掉全世界解释权

工作流：

`outline -> review -> rewrite`

不要误用 `fix`。

如果问题是：

- 章功能错了
- 事件顺序错了
- 角色动力卡本身就不成立

那局部修补只会把坏结构抛光。

## 先看哪几个信号

### 看完正文后，先问这 4 个问题

1. 这章除了主角，还有谁真正主动做事？
2. 支持者帮忙时，是不是也在保自己的东西？
3. 对立方输了，是因为主角更强，还是因为对方也有自己的误判和成本？
4. 这章赢完以后，局面有没有留下更难的残账？

如果四个问题里有两个答不上来，优先走 `review -> fix`。

## 按题材选关注点

### 宫斗 / 宅斗 / 权谋

优先盯：

- 同盟是否单声道
- 家门 / 族面 / 规矩是否真的形成阻力
- 胜利是不是“抢到范围”而不是“一步吃死”

### 现实情感 / 离婚切割

优先盯：

- 现实代价是否真的落在账、房、工作、名声上
- 帮主角的人是不是也在保自己的现实利益
- 前任 / 婆家 /机构是否只是“坏人样本”

### 双女主替身 / 白月光 / 女配路线

优先盯：

- 双核心关系有没有自动回甜
- 白月光是否被写成递刀工具人
- 执行人和协助人是否各自有边界

### 系统 / 穿书 / 游戏

优先盯：

- NPC 是否只在发任务时活着
- 主角的元知识是否太像外挂视角
- 他人命运是否全被压成主角的操作界面

当前项目内已验证：

- [系统任务线交叉验证](./cross-genre-smoke-system-taskline-2026-03-22.md)

## 与 `novel-precheck` 的配合

投稿前别只看平台节奏，也要看结构状态。

推荐规则：

- `precheck = ready-now`
  - 可以进入包装 / 导出
- `precheck = revise-then-submit`
  - 先回正文修，不要拿包装补结构问题

尤其当 `review` 已经给出这些信号时：

- `anti_flattening_flags`
- `recommended_next_action = novel-fix`
- 关系 / 阵营 / 代价维度明显偏低

那就不要跳过修补直接投稿。

## 最实用的路由判断

### 可以直接继续写

满足大多数这些条件时：

- `recommended_next_action = novel-write`
- `anti_flattening_flags = []`
- 当前问题只剩维护提醒
- 字数和节奏没有明显压垮结构

### 应该先修再写

满足任一项就优先 `novel-fix`：

- 主角的信息优势来源发虚
- 对立面的防守动作太弱
- 支持者开始像工具人
- 章内解释句明显吃掉动作张力
- 这章赢得太顺，残账太少

### 应该回去重做大纲

满足任一项就考虑 `novel-rewrite` 或回推 `outline`：

- 角色为什么这样做根本说不圆
- 这章的冲突不是“难打”，而是“根本不存在”
- 任何人都可以被别的功能角色替换

## 当前建议

如果后续继续扩 smoke，优先按这一个方向补样本：

1. `穿书 / 游戏世界关系线`
   - 验证“原书角色 / NPC 是否被压成剧情节点”

## 结论

这份体系现在已经不只是“能发现角色扁平”，而是已经能指导：

- 怎么写
- 怎么审
- 怎么修
- 什么时候继续推进
- 什么时候先停下来补结构

真正好用的地方不在于它多复杂，而在于它已经能把不同流派导向不同但稳定的动作：

- 有的章，直接继续写
- 有的章，必须先 fix
- 有的问题，必须回 outline
