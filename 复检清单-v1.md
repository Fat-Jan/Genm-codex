# 复检清单 v1

## 用途

本清单用于其他模型完成初筛盘点后，由总控模型（我）进行二次审查的标准。

不是改代码，不是改文档，只是判断结论是否站得住。

---

## 0. 复检前提

拿到其他模型交付的三部分内容后，才开始复检：

1. 一张完整清单表
2. 四类候选分组
3. 3 个争议项说明

---

## 1. 复检原则

### 1.1 保守原则

如果拿不准，优先保资产，不优先删。

这意味着：
- `orphan-candidate` 如果有争议，倾向于先降级观察，不倾向于直接判死
- `hard-consumed-candidate` 如果有争议，倾向于保留核心地位

### 1.2 证据优先原则

复检时不看“文件名听起来重不重要”，只看：

- evidence_level 是否和 consumption_guess 匹配
- declaration 和 consumption 是否脱节
- 命中类型权重是否被正确对待

### 1.3 独立判断原则

复检时不依赖其他模型的结论，自己重新推一遍。

---

## 2. 复检检查表

按顺序逐项检查。

---

### 检查 A：evidence_level 和 consumption_guess 是否匹配

这是最容易出错的地方。

#### A1：E3 但给了 `orphan-candidate`

这是矛盾结论。

- E3 代表有 script/test 直接消费
- `orphan-candidate` 代表基本无人消费

两者同时出现，说明要么 evidence_level 打高了，要么 consumption_guess 打低了。

**复检动作**：重新确认 E3 是否成立。

---

#### A2：E2 但给了 `hard-consumed-candidate`

这也是矛盾结论。

- E2 代表只有 SKILL.md / dependency map 声明
- `hard-consumed-candidate` 代表有 script/test 直接强消费

如果只有声明没有脚本，consumption_guess 最多到 `soft-consumed-candidate`。

**复检动作**：再 grep 一次 scripts 和 tests，确认是否有真实消费被漏扫。

---

#### A3：E1 但给了 `soft-consumed-candidate` 或更高

如果只有 docs/findings 提及，但没有 SKILL.md 声明，也没有脚本消费，consumption_guess 最多到 `declared-only-candidate`，通常更像是 `orphan-candidate`。

**复检动作**：确认是否漏掉了 SKILL.md 声明。

---

### 检查 B：`orphan-candidate` 是否可能误伤 soft-consumed

这是最需要仔细看的一类。

#### B1：文档型 / 提示型文件被判 orphan

典型特征：

- 文件是 `.md` 格式
- 主要内容是写作指导、流程说明、checklist
- 在 `SKILL.md` 中被提到"Read"或"参考"

这类很可能是 `soft-consumed`，不是 orphan。

典型误判对象：
- framework README
- QUICK 指南
- writing guidance
- 某些 checkcard

**复检动作**：检查 `SKILL.md` 是否有明确引用。如果有，不应判 orphan。

---

#### B2：checkcard / overlay 类文件被判 orphan

典型特征：

- 名字含 `checkcard`、`overlay`、`guideline`
- 主要被 skill 提示引用
- 目的是给 LLM 补充上下文

这类也倾向于 `soft-consumed`，不是 orphan。

**复检动作**：检查是否有 skill 主流程引用。如果有，倾向于 `soft-consumed-candidate`。

---

### 检查 C：`hard-consumed-candidate` 是否可能过激

这类也容易判错，但不是往删的方向，而是往“给太高”的方向。

#### C1：schema / registry / contract 文件被直接判 hard

典型特征：

- 文件名含 `schema`、`registry`、`contract`
- 被某个脚本读取过一次

但注意：被读取一次不一定等于 hard-consumed。

需要确认：

- 是否被多次消费
- 是否影响校验结果或流程分支
- 还是有读取但从不做判断

如果只是"读了一下但没影响结果"，consumption_guess 应该是 `soft-consumed-candidate` 而不是 `hard-consumed-candidate`。

**复检动作**：查看脚本里是如何使用这个文件的。如果只是 load 但不影响判断，降级为 `soft-consumed-candidate`。

---

#### C2：某个文件被一个脚本 load 后没做任何判断

这也是常见的"假 hard"。

典型模式：
```python
registry = load_json(some_file)
# 读取后没有任何分支判断
```

这种情况应该降级为 `soft-consumed-candidate`。

**复检动作**：看脚本代码里是否有基于该文件内容的 if/分支/校验逻辑。

---

### 检查 D：E2 + 多处声明但无脚本消费

这是"依赖幻觉"最典型的样子，需要重点复检。

特征：

- evidence_level = E2
- `declared_in` 里有多个 SKILL.md 或 dependency map 条目
- `consumed_in` = `none found` 或只有 docs/findings

这种情况：

- 不能因为"被声明很多"就升级为 hard
- 正确的 consumption_guess 是 `declared-only-candidate`
- 这恰恰是本次盘点要识别的那类"幻觉依赖"

**复检动作**：确认没有 script/test 被漏扫。如果确认没有，维持 `declared-only-candidate`。

---

### 检查 E：override 文件是否被正确识别

如果候选文件属于 override 类型（与上游同路径但内容不同），还需要额外检查。

#### E1：是否漏掉了 override 属性

如果一个文件：

- 在 `shared/` 里存在
- 上游也有同路径
- 但本地内容不同

需要额外标注：

- ownership: override
- 并记录 drift 状态

**复检动作**：如果有同路径上游文件，需要确认是否内容不同。

---

### 检查 F：争议项是否真的争议

其他模型列出的 3 个争议项，逐个重新判断。

#### F1：争议理由是否充分

其他模型说"拿不准"，需要看理由是否合理。

如果理由是：

- "文件名看起来很重要" → 这不是合理理由
- "不知道有没有被 script 消费" → 这是合理理由

#### F2：重新独立判断

不复查其他模型的结果，直接用同样的规则重新打一遍结论。

如果结论不同，需要说明原因。

---

## 3. 复检输出格式

复检完成后，输出格式如下：

### 3.1 确认维持

对于其他模型结论正确、无需修改的项：

```
[确认] path → consumption_guess
理由：证据充分，结论合理
```

---

### 3.2 上调

对于需要升级 consumption 的项（例如 orphan → soft）：

```
[上调] path: orphan-candidate → soft-consumed-candidate
理由：E2 成立，SKILL.md 有明确引用，但未见 script 强消费
```

---

### 3.3 下调

对于需要降级 consumption 的项（例如 hard → soft）：

```
[下调] path: hard-consumed-candidate → soft-consumed-candidate
理由：仅有 script load，未做分支判断，未影响流程
```

---

### 3.4 新增 override 标注

对于发现属于 override 的项：

```
[override] path
理由：同路径上游文件存在，本地内容不同
```

---

## 4. 争议升级规则

如果复检后仍拿不准，按以下规则处理：

### 4.1 无法确认是否有 script 消费

保守处理，倾向于维持较高 consumption_guess。

但需要注明：
```
[存疑] path: 无法确认 script 消费，建议补充 grep 验证
```

---

### 4.2 无法确认是否属于 override

先不标注 override，单独记录为待确认项。

---

### 4.3 无法确认是 orphan 还是 soft-consumed

保守处理，倾向于 `soft-consumed-candidate`，不倾向于 `orphan-candidate`。

---

## 5. 复检后的最终交付

复检完成后，输出最终四象限分组：

### 5.1 Core Lane（hard-consumed 确认项）

经复检确认 evidence_level ≥ E3，且 consumption_guess = hard-consumed。

### 5.2 Soft Lane（soft-consumed 确认项）

经复检确认 evidence_level = E2~E3，且 consumption_guess = soft-consumed。

### 5.3 Review Lane（declared-only 确认项）

经复检确认 evidence_level = E2，且 consumption_guess = declared-only。

### 5.4 Orphan Lane（orphan 确认项）

经复检确认 evidence_level ≤ E1，且 consumption_guess = orphan。

---

## 6. 禁止事项

### 禁止 1：不看证据，只看文件名

schema / contract / registry 这些名字不能自动等于 hard-consumed。

### 禁止 2：把"无法确认"等同于"没有"

如果其他模型说"不确定有没有 script 消费"，复检时需要自己再 grep，不能直接当没有处理。

### 禁止 3：把"被多处声明"当作强消费证据

SKILL.md 声明次数多，只能说明"依赖主张多"，不能自动升级 evidence_level。

### 禁止 4：不做交叉验证就下结论

至少用两个独立来源确认 evidence_level，不能只凭一个 grep 结果下结论。

---

## 7. 复检清单速查表

| 场景 | 期望结论 | 如果发现 | 复检动作 |
| --- | --- | --- | --- |
| E3 + orphan-candidate | 矛盾 | 重新确认 E3 是否成立 |
| E2 + hard-consumed-candidate | 矛盾 | 再 grep scripts/tests 确认 |
| E1 + soft-consumed-candidate | 偏高 | 确认 SKILL.md 是否有声明 |
| 文档型 .md 被判 orphan | 可能误判 | 检查 SKILL.md 是否有引用 |
| schema 被判 hard-consumed | 可能偏高 | 检查是否有分支判断影响 |
| E2 + 多处声明 | 典型幻觉依赖 | 维持 declared-only |
| override 文件 | 需额外标注 ownership | 确认同路径上游是否存在 |
