## 文档治理与安全归档方案（规划阶段）

## Goal
在**不影响 runtime / skill / script 行为**的前提下，对仓库做一轮文档治理规划。

本轮只产出：
- 安全文档治理方案
- 可瘦身文档候选清单
- 可归档文档候选清单
- 状态命名 / 分层规则
- 最小安全执行顺序

本轮**不执行**大规模物理迁移、重命名或路径搬运。

## Core Constraints
- 不搬动被 `skills/*/SKILL.md`、`scripts/*`、README、入口文档显式引用的文档路径
- 不创建新的 runtime truth center
- 不把历史记录清理成不可追溯状态
- 优先做“状态归类 + 入口瘦身 + 索引治理”，后做物理归档
- 任何归档动作都必须保留回链入口

## Problems To Solve
1. 入口文档过长，读入成本高
2. 历史内容与当前有效内容混杂
3. 文件状态不透明，文件名/目录名不足以表达是否仍活跃
4. 文档数量增加后，接手者需要频繁点开全文才能判断状态

## Proposed State Model

### A. 不可动（运行时/入口强依赖）
适用对象：
- `README.md`
- `docs/INDEX.md`
- `docs/00-当前有效/*` 中被 README / start-here / skill-usage / skills 显式引用的文档
- `docs/*framework*/README.md` 与 QUICK / rule-cache 入口
- 被 `SKILL.md` 明确写死路径引用的文档

治理动作：
- 可瘦身
- 可补索引
- 可补状态说明
- **不可直接迁移路径**

### B. 可瘦身
适用对象：
- 入口长文档
- 高概率被频繁读取、但同时混有历史增量的文档

治理动作：
- 保留短入口
- 将历史细节拆到附属文档或归档文档
- 维持原路径不变

### C. 可归档候选
适用对象：
- 已完成且不再承担默认入口职责的阶段性总结
- 研究记录中已经收口、不再作为主线依据的内容
- 仅历史参考价值的补充说明

治理动作：
- 进入 `docs/90-归档/` 或保持原处但在索引中标注 archived-candidate
- 必须保留从入口文档或索引的回链

### D. 仅补状态，不搬动
适用对象：
- 仍有参考价值，但当前状态不清的文档
- 未来可能继续使用、但暂时不在默认主线的文档

治理动作：
- 在索引中标注：`active / in-progress / archived / exception / candidate`
- 必要时在文档头部增加一句状态说明

## Proposed Naming / Status Rules

不建议大规模改文件名；优先在索引和文档头部标状态：

- `active`：当前默认入口 / 当前有效规则
- `in-progress`：仍在推进、未收口
- `candidate`：归档候选 / 待整理
- `archived`：已归档，仅历史参考
- `exception`：专项例外，不适用默认规则

建议优先落点：
- `docs/INDEX.md` 统一标注
- 各状态目录下 `README.md` 补一句目录级状态说明
- 少量高价值文档头部补一行 `Status:`

## Minimum Safe Execution Order

1. 先补 `docs/INDEX.md` 的状态标签与候选标识
2. 再瘦身 `README.md` / `start-here.md` / `skill-usage.md`

5. 对任何物理迁移，必须先检查是否存在显式路径引用
