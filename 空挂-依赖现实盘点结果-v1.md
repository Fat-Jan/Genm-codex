## C. 3个最有争议项

### 争议1：`shared/templates/state-v5-template.json`

**为什么拿不准**：
- 一方面 `tests/test_state_contracts.py` 多处直接读取它做校验（属于测试层 E3 消费）
- 另一方面 `scripts/` 层无任何直接读取，它的 `$schema` 指向 `state-schema-v5.json`，但脚本并不通过 schema 做校验
- sync-governance 把它列为保护项，但从消费证据看更像是"声明了重要但实际使用频率偏低"的 Schema Template

**怀疑**：可能属于"Schema 声明重要，但 runtime 实际不直接引用"的 dependency illusion 模式。

---

### 争议2：`shared/references/shared/consumer-read-manifest.md`

**为什么拿不准**：
- 被6个 SKILL.md 声明为"Read first"，声明密度非常高
- 但测试只检查存在性，scripts 层无任何读取
- 实际上 SKILL.md 的"Read"声明更像是给 AI Agent 的提示词引导，不等于文件真的会被 Agent 读取

**怀疑**：SKILL.md 中的"Read xxx"是给 LLM 的指令，可能不会全部执行。这属于"技能叙事声明 vs 实际执行"的典型 gap。

---

### 争议3：`shared/references/shared/core-constraints.md`

**为什么拿不准**：
- 被多个 SKILL.md 声明为 required-like，是清单中声明密度最高的文件之一
- 但 scripts 层无直接消费证据
- 它的内容是写作约束规则，更像是给 AI 看的提示词，Agent 执行 skill 时是否真的读取它存疑

**怀疑**：core-constraints 更像是"共享提示词片段"而不是"被程序消费的资产"，如果 AI 已经把约束内化了，这个文件可能只是"文档备份"而非真实依赖。

---

## 复检重点

1. **`orphan-candidate` 的3个文件**：需防止误伤 `soft-consumed`，特别是 `post-write-validator.md` 可能有历史消费路径
2. **`hard-consumed-candidate` 中的 schema/registry/map 文件**：需防止因文件名（schema/registry/map）听起来重要而误判为强依赖
3. **E2 等级但声明很重的项**：`consumer-read-manifest.md` 和 `core-constraints.md` 最容易属于 dependency illusion
4. **多 SKILL.md 声明但 scripts 不读的项**：如 `consumer-read-manifest.md`，通常最值得重写依赖叙事
